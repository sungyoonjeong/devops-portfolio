# Linux 실전 트러블슈팅 실습 기록

> 작성: 2026-06-21 · 환경: WSL2 (Ubuntu, Linux 6.6.87.2-microsoft-standard-WSL2)  
> 목적: 면접 답변에 "직접 해봤다"를 붙이기 위한 실습. 명령어 암기가 아니라 **왜 이 순서인지** 체득.

## 실습 시나리오 1 — CPU 100% 진단

### 인위적 CPU 부하 생성

```bash
yes > /dev/null &
# [1] 1440
```

`yes` 명령은 "y\n"을 /dev/null에 무한 write한다. 실제 앱에서 무한루프가 발생한 상황을 시뮬레이션한다.

---

### Step 1. top -H -p \<PID\> — 스레드 레벨까지 좁히기

```bash
top -H -p $(pgrep yes)
```

**실제 출력:**
```
top - 23:01:27 up 24 min,  1 user,  load average: 0.33, 0.45, 0.46
Threads:   1 total,   1 running,   0 sleeping,   0 stopped,   0 zombie
%Cpu(s):  5.6 us,  2.8 sy,  0.0 ni, 91.7 id,  0.0 wa,  0.0 hi,  0.0 si,  0.0 st
MiB Mem :   7859.2 total,   6361.5 free,    668.0 used,    829.7 buff/cache
MiB Swap:   2048.0 total,   2048.0 free,      0.0 used.   7034.8 avail Mem

  PID USER      PR  NI    VIRT    RES    SHR S  %CPU  %MEM     TIME+ COMMAND
 1440 jsy       20   0    3208   1792   1792 R  99.9   0.0   0:03.45 yes
```

**해석:**
- PID 1440, 스레드 1개짜리 `yes`가 CPU 99.9% 점유
- `%us` 높음 → 커널 문제 아님, 앱(사용자 코드) 문제
- `wa=0` → I/O 대기 없음. CPU 연산 자체가 문제
- 스레드 1개가 100%면 → 단일 스레드 무한루프 패턴

**면접 포인트:** "`top`으로 PID를 잡은 뒤 `top -H -p`로 어느 스레드가 CPU를 먹는지 스레드 레벨까지 좁힙니다. 멀티스레드 앱이라면 여러 스레드 중 특정 하나가 100%인지 전체가 분산됐는지 여기서 구분됩니다."

---

### Step 2. strace -p \<PID\> -c — 어떤 시스템콜이 폭발하는가

```bash
sudo strace -p 1440 -c
# Ctrl+C로 종료 후 요약 출력
```

**실제 출력:**
```
strace: Process 1440 attached
^Cstrace: Process 1440 detached
% time     seconds  usecs/call     calls    errors syscall
---------- --------- ----------- --------- --------- ----------------
 100.00    8.119451          21    383474           write
---------- --------- ----------- --------- --------- ----------------
 100.00    8.119451          21    383474           total
```

**해석:**
- `write` 시스템콜이 383,474회 — CPU 시간 100% 점유
- 같은 syscall이 수십만 회 반복 → **무한루프** 확정
- 만약 `read`가 반복됐다면 → I/O 대기 (디스크/소켓 블로킹)
- 만약 `futex`가 반복됐다면 → 락 경쟁 (스레드 동기화 문제)

**주의:** `strace`는 ptrace 권한이 필요하다. 타 유저 프로세스에는 `sudo` 필수. `ptrace_scope=1`(WSL 기본값)에서는 자기 자식 프로세스가 아니면 일반 유저로 attach 불가.

**면접 포인트:** "strace로 확인했더니 `write` 시스템콜이 38만 회 반복되고 있었습니다. 같은 syscall이 폭발적으로 반복되면 무한루프가 원인이고, `read` 블로킹이 반복되면 I/O 대기 패턴입니다."

---

### Step 3. ss -tulnp — 소켓·연결 상태 확인

```bash
sudo ss -tulnp
```

**실제 출력:**
```
Netid  State    Recv-Q  Send-Q   Local Address:Port    Peer Address:Port  Process
udp    UNCONN   0       0        127.0.0.53%lo:53      0.0.0.0:*          users:(("systemd-resolve",pid=122,fd=13))
udp    UNCONN   0       0        10.255.255.254:53     0.0.0.0:*
udp    UNCONN   0       0        127.0.0.1:323         0.0.0.0:*
udp    UNCONN   0       0        [::1]:323             [::]:*
tcp    LISTEN   0       1000     10.255.255.254:53     0.0.0.0:*
tcp    LISTEN   0       128      0.0.0.0:22            0.0.0.0:*          users:(("sshd",pid=245,fd=3))
tcp    LISTEN   0       4096     127.0.0.53%lo:53      0.0.0.0:*          users:(("systemd-resolve",pid=122,fd=14))
tcp    LISTEN   0       128      [::]:22               [::]:*             users:(("sshd",pid=245,fd=4))
```

**해석:**
- `sshd` pid=245 → port 22 LISTEN (IPv4 + IPv6)
- `systemd-resolve` pid=122 → port 53 (DNS 리졸버)
- `10.255.255.254:53` → WSL 내부 가상 어댑터, Process 컬럼 비어있음 (정상)
- ESTABLISHED 없음 → 현재 활성 연결 없음
- **sudo 없이 실행하면 Process 컬럼이 비어있음** — 타 유저 프로세스 정보는 root 권한 필요

**CPU 100% 시나리오에서 보는 것:**
```bash
ss -tnp    # LISTEN 제외, 현재 연결된 TCP만
# ESTABLISHED 폭증 → 연결 폭주 or 커넥션 풀 고갈
# CLOSE_WAIT 다수 → 서버가 소켓을 안 닫는 누수
```

**포트 충돌 확인:**
```bash
sudo ss -tulnp | grep :8080    # 8080 점유 프로세스 확인
# ss가 netstat보다 빠르고 현대적 — 면접에서 ss 쓰면 어필
```

### Step 4. lsof -p \<PID\> — 열린 파일·소켓·FD 목록

```bash
sudo lsof -p 1440 | head -20
```

**실제 출력:**
```
COMMAND  PID USER   FD   TYPE DEVICE SIZE/OFF  NODE NAME
yes      995  jsy   cwd  DIR   8,48     4096   2395 /home/jsy/devops-portfolio
yes      995  jsy   rtd  DIR   8,48     4096      2 /
yes      995  jsy   txt  REG   8,48    31112   2248 /usr/bin/yes
yes      995  jsy   mem  REG   8,48  2220400  43579 /usr/lib/x86_64-linux-gnu/libc.so.6
yes      995  jsy   mem  REG   8,48   353616  19098 /usr/lib/locale/C.utf8/LC_CTYPE
yes      995  jsy   mem  REG   8,48       50  19831 /usr/lib/locale/C.utf8/LC_NUMERIC
...
yes      995  jsy    0u  CHR  136,2      0t0      5 /dev/pts/2
```

**해석:**
- `txt` → 실행 바이너리 `/usr/bin/yes`
- `mem` → 메모리에 로드된 공유 라이브러리 (libc 등)
- `0u CHR /dev/pts/2` → 표준입력이 터미널(pts/2)에 연결됨

**실제 진단에서 보는 것 (서버 프로세스):**
```bash
sudo lsof -p <PID> | wc -l       # FD 총 개수
sudo lsof -p <PID> | grep -i sock  # 소켓만 필터
```
FD 수가 수천 개이고 계속 늘어난다 → 소켓/파일을 열고 닫지 않는 **FD 누수**

## 실습 시나리오 2 — 디스크 꽉 참 진단

### Step 1. df -h — 어느 파티션이 꽉 찼나

```bash
df -h
```

**실제 출력:**
```
Filesystem      Size  Used Avail Use% Mounted on
/dev/sdd        1007G  18G  939G   2% /
C:\              931G 383G  548G  42% /mnt/c
snapfuse          50M  50M     0 100% /snap/snapd/26865
snapfuse          74M  74M     0 100% /snap/core22/2411
snapfuse          42M  42M     0 100% /snap/trivy/276
```

**해석:**
- `/dev/sdd` (WSL 루트) → 2% 사용, 여유 충분
- `/mnt/c` → Windows C드라이브 마운트, 42%
- `snapfuse 100%` → snap 패키지는 읽기 전용 squashfs 이미지라 **항상 100% 표시됨. 무시해도 됨.**

### Step 2. du -sh /var/log/* — 범인 디렉터리 좁히기

```bash
du -sh /var/log/*
```

**실제 출력 (주요 항목):**
```
809M    /var/log/journal      ← 압도적 1위 (범인)
2.3M    /var/log/syslog.1
1.2M    /var/log/kern.log.1
804K    /var/log/dpkg.log.1
460K    /var/log/syslog
300K    /var/log/dpkg.log
236K    /var/log/apt
208K    /var/log/kern.log
128K    /var/log/wtmp
108K    /var/log/bootstrap.log
```

**해석:**
- `/var/log/journal` 809MB — systemd 로그 저장소. 오래 쌓이면 GB 단위로 커짐
- 실제 서버에서 디스크 꽉 찼을 때 `/var/log/journal`이 범인인 경우가 매우 흔함

**주의:** `sudo du -ah / | sort -rh | head -20`을 WSL에서 실행하면 `/mnt/c`의 Windows 보호 디렉터리들에서 Permission denied가 폭발함. 범위를 좁혀서:
```bash
sudo du -ah /var /home /tmp 2>/dev/null | sort -rh | head -20
```

### Step 3. 조치 — journalctl vacuum

```bash
journalctl --disk-usage        # 현재 journal 사용량 확인
```

## 실습 시나리오 3 — 메모리 / I/O 상태 확인

### free -h — 메모리 전체 현황

```bash
free -h
```

**실제 출력:**
```
              total        used        free      shared  buff/cache   available
Mem:          7.7Gi       676Mi       6.0Gi       3.0Mi       1.0Gi       6.9Gi
Swap:         2.0Gi          0B       2.0Gi
```

**해석:**
- 메모리 7.7G 중 676M 사용 → 여유 충분
- Swap used=0B → 스왑 사용 없음, 메모리 압박 없음
- `available 6.9Gi` → 실제로 쓸 수 있는 메모리 (buff/cache는 필요하면 반납)

**OOM 직전 징조:** `free` 출력에서 available이 거의 0에 수렴하고 Swap used가 급증하면 위험 신호.

---

### vmstat 1 5 — 메모리·스왑·I/O·CPU 종합

```bash
vmstat 1 5    # 1초 간격으로 5회 출력
```

**실제 출력:**
```
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in    cs us sy id wa st
 0  0      0 6517128 77196 791004    0    0    54     3  220   408  3  3 94  0  0
 0  0      0 6517128 77196 791044    0    0     0   177  281     0  0  0 100  0  0
 0  0      0 6517128 77196 791044    0    0     0     0  153   264  0  0 100  0  0
 0  0      0 6516876 77196 791044    0    0     0     0  796  1464  0  0  99  0  0
 0  0      0 6515920 77196 791044    0    0     0     0  190   341  0  0 100  0  0
```

**컬럼별 해석:**

| 컬럼 | 값 | 해석 |
|------|-----|------|
| r | 0 | 실행 대기 프로세스 없음 |
| b | 0 | I/O 블로킹 프로세스 없음 |
| swpd | 0 | 스왑 사용량 0 |
| si/so | 0/0 | 스왑 in/out 없음 → Thrashing 아님 |
| wa | 0 | CPU I/O 대기 없음 → 디스크 병목 없음 |
| id | 94~100 | CPU 거의 유휴 상태 |

> 첫 번째 행은 **부팅 이후 평균값** (실시간 아님). `bi=54`는 역대 평균이고 현재 I/O는 0.

**위험 신호 패턴:**
```
si/so 지속적으로 0 아님  → Thrashing (스왑 폭주)
wa 10 이상              → 디스크/네트워크 I/O 병목
r 값이 CPU 수보다 큼     → CPU 과포화 (run queue 적체)
```

### iostat -x 1 — 디스크 I/O 병목 상세

```bash
iostat -x 1    # sysstat 패키지 필요: sudo apt install sysstat
```

**실제 출력 (주요 디바이스):**
```
avg-cpu: %user  %nice  %system  %iowait  %steal  %idle
          2.22   0.00     2.59     0.04    0.00  95.15

Device   r/s   rkB/s  rrqm/s  %rrqm  r_await  rareq-sz
         w/s   wkB/s  wrqm/s  %wrqm  w_await  wareq-sz   %util

sdd      9.49  490.88   4.57   32.51    0.42    51.72
         2.36   42.80   3.54   60.00    2.95    18.12    0.34
```

**핵심 컬럼:**
- `%util=0.34` → 디스크 사용률 0.34%. 병목 없음
- `%wrqm=60.00` → 쓰기 요청 병합률 60% (journal 정리 직후 쓰기 발생)
- `r_await / w_await` → 요청 대기 시간(ms). 높을수록 병목

**위험 신호:** `%util=100%` 근처이면서 `await`가 높으면 디스크 포화.

### dmesg | grep -i "out of memory" — OOM Killer 발동 이력

```bash
dmesg | grep -i "out of memory"
# (출력 없음 — OOM 발생 이력 없음)
```

**해석:** 아무것도 안 나오면 정상. OOM Killer가 발동됐다면:
```
[123456.789] Out of memory: Kill process 1234 (java) score 987 or sacrifice child
```
이런 라인이 찍힌다. 어떤 프로세스가 얼마(score)의 메모리 압박으로 종료됐는지 확인 가능.

```bash
# 추가 확인
cat /var/log/syslog | grep OOM
```

```
    → futex 반복 → 락 경쟁
         │
         ▼
[4] sudo ss -tulnp
    → ESTABLISHED 폭증 → 연결 폭주

    → FD 수가 수천 개이고 증가 중 → FD 누수
         │

    I/O 대기  → iostat으로 병목 디스크 확인, DB slow query 점검

```

## 명령어 빠른 참조

```bash
# CPU
top                          # 전체 프로세스 CPU 순 정렬
top -H -p <PID>              # 특정 PID 스레드 레벨
sudo strace -p <PID> -c      # syscall 카운트 요약 (Ctrl+C로 종료)

# 메모리
free -h                      # 전체 메모리·스왑 현황
vmstat 1 5                   # 메모리·스왑·I/O·CPU 종합 (1초 간격 5회)
cat /proc/<PID>/status       # 특정 프로세스 메모리 상세 (VmRSS, VmSwap)
dmesg | grep -i "out of memory"  # OOM Killer 이력

# 디스크
df -h                        # 파티션별 사용량
du -sh /var/log/*            # 디렉터리별 용량
sudo du -ah /var /home 2>/dev/null | sort -rh | head -20  # 큰 파일 상위
iostat -x 1                  # 디스크 I/O 병목 (%util, await)
iotop                        # 어느 프로세스가 I/O를 쓰는지 실시간

# 소켓·네트워크
sudo ss -tulnp               # LISTEN 포트 + 프로세스
ss -tnp                      # 현재 연결된 TCP (ESTABLISHED)
sudo ss -tulnp | grep :8080  # 특정 포트 점유 확인
sudo lsof -p <PID>           # 프로세스 열린 파일·소켓·FD

# 로그 정리
sudo journalctl --vacuum-time=2weeks   # 오래된 journal 삭제
sudo journalctl --vacuum-size=100M     # 크기 제한
```

## 면접 2분 답변 — CPU 100% 진단

> "먼저 `top`으로 CPU를 잡아먹는 PID를 특정하고, `top -H -p`로 어느 스레드인지 스레드 레벨까지 좁힙니다. `strace -c`로 어떤 시스템콜이 반복되는지 보면 무한루프(write 폭발)인지 I/O 대기(read 블로킹)인지 구분됩니다. `ss`로 소켓 상태에서 연결 폭주가 없는지, `lsof`로 FD 누수가 없는지 확인하고, `vmstat`으로 스왑까지 종합 확인합니다. WSL 실습에서 `yes > /dev/null`로 CPU 100% 상황을 만들어 strace로 write 38만 회 반복을 직접 확인했습니다."
