## 1. Docker란 무엇인가

```
Docker = 애플리케이션을 "컨테이너"라는 격리된 환경에
패키징해서 실행하는 플랫폼

핵심 가치:
  "한 번 빌드하면 어디서나 동일하게 실행"
  → 내 컴퓨터에서 되던 게 서버에서 안 되는 문제 해결

DevOps에서 왜 중요한가:
  CI/CD 파이프라인의 핵심 단위
  K8s가 관리하는 가장 작은 배포 단위
  마이크로서비스 아키텍처의 기본 구성요소
```

## 2. 컨테이너 vs 가상머신(VM)

```
가상머신(VM):
  하드웨어 위에 하이퍼바이저
  → 그 위에 OS 전체(Guest OS)를 통째로 설치
  → 무겁고 부팅 느림 (수 분)
  → 격리성은 매우 높음

컨테이너:
  호스트 OS의 커널을 공유
  → 프로세스 단위로 격리 (Namespace·cgroup 기술)
  → 가볍고 빠름 (수 초)
  → VM보다 격리성은 낮지만 충분함
```

```
시각화:

[VM 구조]
App A | App B | App C
Guest OS | Guest OS | Guest OS
------- Hypervisor -------
       Host OS
       Hardware

[컨테이너 구조]
App A | App B | App C
------- Docker Engine -------
       Host OS
       Hardware
```

| 비교 | VM | 컨테이너 |
|------|----|---------|
| 부팅 시간 | 분 단위 | 초 단위 |
| 크기 | GB 단위 | MB 단위 |
| 격리 수준 | 매우 높음 (OS 단위) | 프로세스 단위 |
| 리소스 효율 | 낮음 | 높음 |

## 3. Docker 아키�처

```
Docker Client (docker 명령어)
      ↓ (REST API)
Docker Daemon (dockerd)
  ├─ 이미지 관리
  ├─ 컨테이너 관리
  ├─ 네트워크 관리
  └─ 볼륨 관리
      ↓
Docker Registry (Docker Hub 등)
  → 이미지를 저장·공유하는 저장소
```

### 핵심 용어 정리

| 용어 | 설명 |
|------|------|
| **이미지(Image)** | 컨테이너 실행을 위한 읽기 전용 템플릿 (클래스 같은 개념) |
| **컨테이너(Container)** | 이미지를 실행한 인스턴스 (객체 같은 개념) |
| **Dockerfile** | 이미지를 만드는 설계도(레시피) |
| **레지스트리(Registry)** | 이미지 저장소 (Docker Hub, ECR 등) |
| **레이어(Layer)** | 이미지를 구성하는 각 단계별 변경사항 |

## 4. 이미지(Image)

### 이미지 다루기

```bash
# 이미지 검색 (Docker Hub)
docker search nginx

# 이미지 다운로드
docker pull nginx
docker pull nginx:1.25       # 특정 버전(태그) 지정
docker pull ubuntu:22.04

# 이미지 목록 확인
docker images
docker image ls

# 이미지 상세 정보
docker inspect nginx

# 이미지 삭제
docker rmi nginx
docker rmi $(docker images -q)   # 전체 이미지 삭제
```

### 이미지 레이어 구조

```
각 명령어(FROM, RUN, COPY...)마다 레이어 생성
레이어는 캐싱됨 → 변경 없는 레이어는 재사용 (빌드 속도 ↑)

예시:
  FROM ubuntu:22.04      ← Layer 1
  RUN apt update          ← Layer 2
  RUN apt install python3 ← Layer 3
  COPY app.py /app/       ← Layer 4

→ app.py만 변경 시 Layer 1~3은 캐시 재사용
→ Layer 4만 다시 빌드 (빌드 속도 향상)
```

### 태그(Tag)

```bash
# 태그 형식: 이미지명:태그
nginx:latest    # 태그 생략 시 latest
nginx:1.25
nginx:1.25-alpine   # alpine = 경량 리눅스 기반

# alpine 이미지를 선호하는 이유
#   일반 이미지: 수백 MB
#   alpine 기반: 수십 MB (훨씬 가벼움)
```

## 5. 컨테이너(Container) 실행·관리

### 컨테이너 실행

```bash
# 기본 실행
docker run nginx

# 백그라운드 실행 (-d: detached)
docker run -d nginx

# 이름 지정 (--name)
docker run -d --name my-nginx nginx

# 포트 매핑 (-p 호스트포트:컨테이너포트)
docker run -d -p 8080:80 nginx
# → localhost:8080 접속 시 컨테이너의 80포트로 연결

# 인터랙티브 모드 (-it: 터미널 접속)
docker run -it ubuntu bash
# -i: 표준입력 유지, -t: 가상 터미널 할당

# 실행 후 자동 삭제 (--rm)
docker run --rm -it ubuntu bash
# 컨테이너 종료 시 자동 삭제 (테스트용으로 유용)
```

### 컨테이너 상태 관리

```bash
# 실행 중인 컨테이너 목록
docker ps

# 전체 컨테이너 목록 (중지된 것 포함)
docker ps -a

# 컨테이너 시작/중지/재시작
docker start my-nginx
docker stop my-nginx
docker restart my-nginx

# 컨테이너 삭제
docker rm my-nginx
docker rm -f my-nginx    # 실행 중이어도 강제 삭제

# 중지된 컨테이너 전체 삭제
docker container prune
```

### 컨테이너 내부 접속

```bash
# 실행 중인 컨테이너에 접속
docker exec -it my-nginx bash
# exec: 이미 실행 중인 컨테이너에서 새 명령 실행

# run vs exec 차이
#   run:  새 컨테이너를 만들어서 실행
#   exec: 이미 실행 중인 컨테이너에 명령 전달
```

## 6. Dockerfile

### 기본 구조

```dockerfile
# 베이스 이미지 지정 (필수, 첫 줄)
FROM python:3.11-slim

# 작업 디렉토리 설정 (이후 명령어의 기준 경로)
WORKDIR /app

# 파일 복사 (호스트 → 컨테이너)
COPY requirements.txt .

# 명령어 실행 (이미지 빌드 시점에 실행)
RUN pip install -r requirements.txt

# 나머지 소스 코드 복사
COPY . .

# 컨테이너 외부에 노출할 포트 (문서화 목적, 실제 매핑은 -p로)
EXPOSE 5000

# 환경변수 설정
ENV FLASK_ENV=production

# 컨테이너 시작 시 실행할 명령 (컨테이너 실행 시점)
CMD ["python", "app.py"]
```

### 주요 명령어 정리

| 명령어 | 시점 | 설명 |
|--------|------|------|
| `FROM` | 빌드 | 베이스 이미지 지정 (필수) |
| `WORKDIR` | 빌드+실행 | 작업 디렉토리 설정 |
| `COPY` | 빌드 | 파일 복사 (호스트→이미지) |
| `ADD` | 빌드 | COPY + URL·압축파일 자동해제 (특수한 경우만) |
| `RUN` | 빌드 | 빌드 시점에 명령 실행 (레이어 생성) |
| `ENV` | 빌드+실행 | 환경변수 설정 |
| `EXPOSE` | 문서화 | 사용 포트 명시 (실제 매핑 아님) |
| `CMD` | 실행 | 컨테이너 시작 시 기본 명령 (덮어쓰기 가능) |
| `ENTRYPOINT` | 실행 | 컨테이너의 고정 실행 명령 (덮어쓰기 어려움) |
| `USER` | 빌드+실행 | 실행 사용자 지정 (보안) |
| `ARG` | 빌드 | 빌드 시점에만 사용하는 변수 |

### CMD vs ENTRYPOINT

```dockerfile
# CMD: 기본값, docker run 시 덮어쓰기 가능
CMD ["python", "app.py"]
# docker run myimage python other.py  → other.py 실행됨

# ENTRYPOINT: 고정, 인자만 추가 가능
ENTRYPOINT ["python"]
CMD ["app.py"]
# docker run myimage  → python app.py
# docker run myimage other.py → python other.py
```

### COPY 시 .dockerignore

```
# .dockerignore 파일 (Dockerfile과 같은 위치)
__pycache__
*.pyc
.git
.env
node_modules

# 이유: 불필요한 파일이 이미지에 포함되면
#   이미지 크기 증가 + 빌드 시간 증가 + 민감정보 노출 위험
```

## 7. 이미지 빌드·태그·푸시

### 빌드

```bash
# 현재 디렉토리(.)의 Dockerfile로 빌드
docker build -t myapp .

# 태그 지정
docker build -t myapp:1.0 .

# 다른 Dockerfile 사용
docker build -f Dockerfile.prod -t myapp:prod .

# 캐시 없이 빌드 (전체 재빌드)
docker build --no-cache -t myapp .
```

### 태그·푸시 (Docker Hub)

```bash
# Docker Hub 로그인
docker login

# 이미지 태그 변경 (사용자명/이미지명:태그)
docker tag myapp:1.0 sungyoonjeong/myapp:1.0

# 푸시 (업로드)
docker push sungyoonjeong/myapp:1.0

# 다른 환경에서 다운로드
docker pull sungyoonjeong/myapp:1.0
```

## 8. 볼륨(Volume) — 데이터 영속성

### 왜 필요한가

```
컨테이너는 삭제되면 내부 데이터도 함께 사라짐
→ DB 데이터, 로그 파일 등은 보존해야 함
→ 볼륨으로 컨테이너 외부에 데이터 저장
```

### 볼륨 종류 3가지

```bash
# ① Named Volume (Docker가 관리하는 영역)
docker volume create mydata
docker run -v mydata:/var/lib/mysql mysql

# ② Bind Mount (호스트의 특정 경로 직접 연결)
docker run -v /home/jsy/data:/var/lib/mysql mysql
docker run -v $(pwd)/logs:/app/logs myapp
# 개발 중 코드 변경을 즉시 반영할 때 유용

# ③ tmpfs (메모리에만 저장, 컨테이너 종료 시 삭제)
docker run --tmpfs /app/cache myapp
```

### 볼륨 관리 명령어

```bash
docker volume ls              # 볼륨 목록
docker volume inspect mydata  # 상세 정보
docker volume rm mydata       # 삭제
docker volume prune           # 사용 안 하는 볼륨 전체 삭제
```

| 종류 | 위치 | 용도 |
|------|------|------|
| Named Volume | Docker 관리 영역 | DB 데이터 등 영속 데이터 |
| Bind Mount | 호스트 지정 경로 | 개발 중 코드 동기화 |
| tmpfs | 메모리 | 임시·민감 데이터 |

## 9. 네트워크(Network)

### 네트워크 드라이버 종류

```bash
docker network ls

# bridge (기본값): 컨테이너 간 격리된 내부망
# host: 호스트 네트워크 직접 사용 (격리 없음)
# none: 네트워크 없음
```

### 컨테이너 간 통신

```bash
# 사용자 정의 네트워크 생성
docker network create mynet

# 같은 네트워크에 컨테이너 연결
docker run -d --name db --network mynet mysql
docker run -d --name app --network mynet myapp

# app 컨테이너에서 db에 접근할 때
#   IP 대신 컨테이너 이름으로 접근 가능
#   예: mysql://db:3306
```

```
핵심:
  같은 네트워크에 있으면
  컨테이너 이름 = 호스트명(DNS)으로 서로 통신 가능
  → docker-compose에서 자동으로 이렇게 구성됨
```

### 네트워크 관리

```bash
docker network create mynet
docker network connect mynet 컨테이너명    # 기존 컨테이너 연결
docker network disconnect mynet 컨테이너명
docker network rm mynet
docker network inspect mynet   # 연결된 컨테이너 확인
```


```python
# Python에서 환경변수 읽기
import os
```

```
  → docker run -e 또는 .env 파일(+ .gitignore) 사용
```

## 11. docker-compose 기초

### compose란

```
여러 컨테이너를 하나의 YAML 파일로 정의하고
한 번에 실행·관리하는 도구

예: 웹앱 + DB + Redis 를 docker run 3번 하는 대신
   docker-compose up 한 번으로 전체 실행
```

### 기본 구조 (docker-compose.yml)

```yaml
version: '3.8'

services:
  web:
    build: .                  # 현재 디렉토리 Dockerfile로 빌드
    ports:
      - "5000:5000"
    environment:
      - DB_HOST=db
    depends_on:
      - db                     # db 먼저 실행 후 web 실행
    volumes:
      - .:/app                 # 코드 동기화

  db:
    image: mysql:8.0
    environment:
      - MYSQL_ROOT_PASSWORD=password
      - MYSQL_DATABASE=mydb
    volumes:
      - dbdata:/var/lib/mysql  # named volume

volumes:
  dbdata:                       # named volume 선언
```

### compose 명령어

```bash
docker-compose up              # 실행 (포그라운드)
docker-compose up -d           # 백그라운드 실행
docker-compose down            # 중지 + 컨테이너·네트워크 삭제
docker-compose down -v         # + 볼륨까지 삭제

docker-compose ps              # 상태 확인
docker-compose logs            # 전체 로그
docker-compose logs -f web     # web 서비스 로그 실시간

docker-compose build           # 이미지만 다시 빌드
docker-compose up --build      # 빌드 후 실행

docker-compose exec web bash   # web 컨테이너 접속
```

## 12. docker-compose 실전

### 핵심 키워드 정리

| 키워드 | 설명 |
|--------|------|
| `services` | 실행할 컨테이너들 정의 |
| `build` | Dockerfile 경로 (이미지 빌드) |
| `image` | 사용할 이미지 (Hub에서 받아옴) |
| `ports` | 포트 매핑 ("호스트:컨테이너") |
| `environment` | 환경변수 |
| `depends_on` | 시작 순서 의존성 |
| `volumes` | 볼륨 마운트 |
| `networks` | 네트워크 설정 (기본: 자동 생성) |
| `restart` | 재시작 정책 (`always`, `on-failure`, `unless-stopped`) |

### depends_on의 한계 (중요)

```yaml
depends_on:
  - db
# 이건 "db 컨테이너가 시작된 후" web을 실행한다는 뜻
# db 컨테이너의 "MySQL 서비스가 준비 완료"를 보장하지 않음!

# 해결: healthcheck 추가
services:
  db:
    image: mysql:8.0
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 5s
      retries: 10

  web:
    depends_on:
      db:
        condition: service_healthy   # db가 healthy할 때까지 대기
```

## 13. 로그·디버깅

```bash
# 컨테이너 로그 확인
docker logs my-nginx
docker logs -f my-nginx        # 실시간 추적 (follow)
docker logs --tail 100 my-nginx # 마지막 100줄

# 컨테이너 리소스 사용량 (실시간)
docker stats

# 컨테이너 프로세스 확인
docker top my-nginx

# 컨테이너 상세 정보 (네트워크·볼륨·환경변수 등 전부)
docker inspect my-nginx

# 컨테이너와 호스트 간 파일 복사
docker cp my-nginx:/etc/nginx/nginx.conf ./nginx.conf
docker cp ./app.py my-nginx:/app/app.py
```

### 컨테이너가 바로 종료될 때 디버깅

```bash
# 컨테이너가 즉시 Exit되는 경우
docker run myapp        # Exit (이유 모름)

# 1. 로그 확인
docker logs $(docker ps -lq)   # 가장 최근 컨테이너 로그

# 2. 컨테이너 안 들어가서 직접 실행해보기
docker run -it myapp bash
# 컨테이너 안에서 CMD에 적은 명령을 직접 실행해보며 에러 확인
```

## 14. 리소스 제한

```bash
# 메모리 제한
docker run -d --memory="512m" myapp

# CPU 제한 (1코어의 50%)
docker run -d --cpus="0.5" myapp

# 메모리 + CPU 동시 제한
docker run -d --memory="512m" --cpus="1.0" myapp
```

```yaml
# docker-compose에서
services:
  web:
    deploy:
      resources:
        limits:
          memory: 512M
          cpus: "0.5"
```

```
DevOps에서 중요한 이유:
  하나의 컨테이너가 메모리를 무한정 사용하면
  같은 서버의 다른 컨테이너에 영향
  → 리소스 제한은 K8s의 requests/limits와 동일한 개념
  → K8s 학습 시 그대로 연결됨
```

## 16. 실전 예제: Python 앱 컨테이너화

### 프로젝트 구조

```
myapp/
├── app.py
├── requirements.txt
├── Dockerfile
├── .dockerignore
└── docker-compose.yml
```

### app.py

```python
from flask import Flask
import os

app = Flask(__name__)

@app.route('/')
def hello():
    env = os.environ.get('APP_ENV', 'development')
    return f"Hello from Docker! Environment: {env}"

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000)
```

### Dockerfile

```dockerfile
FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

ENV APP_ENV=production
EXPOSE 5000

CMD ["python", "app.py"]
```

### .dockerignore

```
__pycache__
*.pyc
.git
.env
```

### docker-compose.yml

```yaml
version: '3.8'

services:
  web:
    build: .
    ports:
      - "5000:5000"
    environment:
      - APP_ENV=development
    volumes:
      - .:/app
```

### 실행

```bash
docker-compose up --build

# 브라우저: localhost:5000
# → "Hello from Docker! Environment: development"
```

## 17. 자주 하는 실수

```bash
# 1. host='0.0.0.0' 빠짐
app.run(host='127.0.0.1')  # ❌ 컨테이너 외부에서 접근 불가
app.run(host='0.0.0.0')    # ✅ 모든 인터페이스에서 접근 허용

# 2. 포트 매핑 방향 혼동
-p 8080:5000   # 호스트:컨테이너 (호스트 8080 → 컨테이너 5000)

# 3. 이미지 빌드 후 코드 수정했는데 반영 안 됨
# → 재빌드 필요
docker build -t myapp .   # 다시 빌드해야 COPY 내용 갱신

# 4. COPY . . 시 .dockerignore 없이 전체 복사
# → node_modules, .git 등 불필요한 파일까지 포함 → 이미지 비대

# 5. 컨테이너 삭제했는데 볼륨 데이터도 사라짐
docker rm -v my-db   # -v 옵션은 볼륨도 함께 삭제하므로 주의

# 6. latest 태그만 사용
# → 나중에 이미지가 바뀌어도 latest는 그대로 가리킬 수 있어
#    "어떤 버전이 배포됐는지" 추적 어려움
# → 운영 환경은 명시적 버전 태그 사용 (v1.2.0 등)
```

## 18. 핵심 명령어 치트시트

| 분류 | 명령어 | 설명 |
|------|--------|------|
| 이미지 | `docker pull/images/rmi` | 다운로드/목록/삭제 |

| 제어 | `docker start/stop/restart/rm` | 컨테이너 제어 |
| 접속 | `docker exec -it name bash` | 실행중 컨테이너 접속 |
| 로그 | `docker logs -f name` | 로그 실시간 확인 |
| 볼륨 | `docker volume create/ls/rm` | 볼륨 관리 |
| 네트워크 | `docker network create/ls` | 네트워크 관리 |
| compose | `docker-compose up -d / down` | 다중 컨테이너 실행/종료 |
