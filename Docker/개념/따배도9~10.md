# [따배도] 3장. 컨테이너 속속들이 살펴보기

## 1. [이론] 컨테이너와 이미지의 이해
컨테이너는 단순한 가상화 기술을 넘어 하나의 완전한 애플리케이션으로 동작합니다.

*   **컨테이너의 정의**: 하나의 독립된 공간에서 실행되는 애플리케이션입니다. 노드 js, PHP, MySQL 등 각각의 컨테이너는 서로 완전하게 격리되어 운영됩니다.
*   **독립성(Isolation)**: 각 컨테이너는 CPU, 메모리, 네트워크, 디스크 리소스를 별도로 할당받으며, 사용자(UID), 호스트 이름 등도 독립적으로 가집니다. 덕분에 한 컨테이너의 변경이 다른 컨테이너에 영향을 주지 않습니다.
*   **도커 호스트(Docker Host)**: 도커 데몬이 실행 중인 리눅스 커널 기반의 시스템을 말하며, 컨테이너가 실행될 수 있는 플랫폼 역할을 합니다. 모든 컨테이너는 호스트의 커널을 공유하지만, 프로세스 관점에서는 독립된 애플리케이션으로 관리됩니다.
*   **레이어 구조(Layered Structure)**:
    *   **베이스 이미지**: OS나 런타임 환경(예: Node.js)을 담은 하위 레이어입니다.
    *   **소스 이미지**: 실제 애플리케이션 코드(예: 앱 소스)를 담은 레이어입니다.
    *   **실행 설정**: 애플리케이션을 어떻게 실행할지에 대한 명령이 포함됩니다.
    *   이미지는 이러한 여러 개의 레이어가 쌓여서 만들어지며, 각 레이어는 고유한 ID로 관리됩니다.

## 2. [이론] 이미지 vs 컨테이너
*   **컨테이너 이미지**: 하드디스크에 파일 형태로 저장된 상태이며, 읽기 전용(Read-Only)입니다.
*   **컨테이너**: 이미지를 실행하여 메모리상에서 프로세스로 동작하는 상태입니다. 이때부터 읽기 및 쓰기(Read-Write)가 가능해집니다.

## 3. [실습] 도커 컨테이너 생명주기 및 주요 명령어
실습을 통해 이미지를 검색하고, 내려받아 실행하고 삭제하는 전체 과정을 확인합니다.

*   **실습 준비**: Ubuntu 환경에서 루트(root) 계정으로 진행하며, `systemctl status docker`로 데몬 실행 여부를 먼저 확인합니다.
*   **이미지 검색 및 다운로드**:
    *   `docker search enginex`: 도커 허브에서 이미지를 검색합니다.
    *   `docker pull nginx`: 이미지를 로컬로 내려받습니다. 이때 이미지 구성 요소인 여러 레이어가 차례로 다운로드됩니다.
    *   다운로드된 레이어 파일은 `/var/lib/docker/overlay2` 디렉토리에서 확인할 수 있습니다.
*   **컨테이너 실행 및 확인**:
    *   `docker run -d --name web -p 80:80 nginx`: 이미지를 'web'이라는 이름의 컨테이너로 백그라운드에서 실행하며, 호스트의 80번 포트와 연결합니다.
    *   `docker ps`: 현재 실행 중인 컨테이너 목록과 상태를 확인합니다.
    *   `curl localhost:80`: 웹 서버가 정상 동작하여 "Welcome to nginx" 페이지를 응답하는지 테스트합니다.
*   **정리 및 삭제**:
    *   `docker stop web`: 실행 중인 컨테이너를 중지합니다.
    *   `docker rm web`: 중지된 컨테이너를 삭제합니다.
    *   `docker rmi nginx`: 로컬에 저장된 컨테이너 이미지를 삭제하여 디스크 공간을 확보합니다.
