# [따배도] 7장. 컨테이너 리소스 관리 및 모니터링

## 1. [이론] 컨테이너 리소스 제한의 필요성
도커 호스트의 하드웨어 리소스(CPU, 메모리, 디스크)는 유한합니다. 별도의 제한을 걸지 않으면 하나의 컨테이너가 호스트의 모든 자원을 점유하여 다른 컨테이너의 동작을 방해할 수 있으므로, 서비스의 안정성을 위해 적절한 쿼터(Quota) 설정이 필수적입니다.

### 주요 리소스 제한 방법
*   **메모리 제한**:
    *   `--memory (-m)`: 컨테이너가 사용할 최대 메모리를 지정합니다. 초과 시 컨테이너가 킬(Kill)될 수 있습니다.
    *   `--memory-reservation`: 최소한으로 보장받는 메모리 양(Soft limit)입니다.
    *   `--memory-swap`: 메모리와 스왑의 합계입니다. (실제 스왑 양 = 스왑 설정값 - 메모리 설정값)
    *   `--oom-kill-disable`: 메모리 부족 시 커널이 컨테이너를 강제로 죽이는 것을 방지합니다.
*   **CPU 제한**:
    *   `--cpus`: 사용할 CPU 코어 수를 지정합니다 (예: `0.5`는 반 개 코어 사용).
    *   `--cpuset-cpus`: 특정 CPU 코어(0번, 1번 등)를 지정하여 할당합니다.
    *   `--cpu-shares`: 컨테이너 간 CPU 사용 가중치를 설정합니다 (기본값 1024).
*   **블록 I/O(디스크) 제한**:
    *   `--blkio-weight`: 디스크 I/O 가중치를 설정합니다 (10~1000).
    *   `--device-read-bps` / `--device-write-bps`: 초당 읽고 쓰는 바이트 수를 제한합니다.
    *   `--device-read-iops` / `--device-write-iops`: 초당 I/O 횟수를 제한합니다.

## 2. [실습] 리소스 관리 및 모니터링 도구 활용
실습에서는 `stress` 프로그램을 사용하여 인위적으로 부하를 일으키고 설정값이 잘 적용되는지 확인합니다.

### 리소스 제한 실습
*   **메모리**: `-m 100m`으로 설정한 컨테이너에 150m의 부하를 주면 컨테이너가 즉시 종료되는 것을 확인할 수 있습니다.
*   **CPU**: `htop` 툴을 통해 특정 코어(`--cpuset-cpus`)만 100% 점유하거나, 가중치(`--cpu-shares`)에 따라 CPU 점유율이 2:1:0.5 비율로 나뉘는 것을 실시간으로 관찰합니다.
*   **디스크**: `dd` 명령어를 사용하여 디스크 쓰기 속도가 설정한 I/O 쿼터에 맞춰 제한되는지 테스트합니다.

### 모니터링 도구
*   **docker stats**: 실행 중인 컨테이너의 CPU, 메모리, 네트워크, I/O 사용량을 실시간 텍스트로 보여줍니다.
*   **docker event**: 도커 호스트의 실시간 event 정보를 수집해서 출력
    - docker events -f container=<NAME>
    - docker image -f container=<NAME>
*   **cAdvisor**: 구글에서 만든 컨테이너 모니터링 툴로, 웹 UI를 통해 호스트와 컨테이너의 자원 사용 현황을 시각적으로 파악할 수 있습니다. 이는 나중에 쿠버네티스 환경에서도 활용됩니다.

# [따배도] 8장. 도커 컨테이너 스토리지 (Storage)

## 1. [이론] 컨테이너 볼륨의 이해와 필요성
도커 컨테이너는 기본적으로 데이터를 영구적으로 보존하지 않는 구조를 가지고 있습니다.

*   **이미지와 컨테이너의 계층 구조**:
    *   **도커 이미지**: 읽기 전용(Read-Only) 속성을 가집니다.
    *   **컨테이너**: 이미지를 실행하면 그 위에 읽기 및 쓰기(Read-Write)가 가능한 레이어가 추가됩니다.
    *   **유니온 파일 시스템(Union File System)**: 읽기 전용 레이어와 쓰기 가능 레이어를 하나로 합쳐서 보여주는 기술입니다(오버레이라고도 함).
*   **데이터 소실 문제**: 컨테이너가 삭제(`docker rm`)되면 해당 컨테이너의 쓰기 레이어도 함께 사라지기 때문에, 그 안에 저장된 데이터(DB 데이터, 로그 등)를 복구할 수 없습니다.
*   **해결책 (볼륨 마운트)**: 호스트 시스템의 특정 저장 공간(디렉토리)을 컨테이너와 연결하여, 컨테이너가 삭제되어도 데이터는 호스트에 남도록 설정하는 기술입니다.

## 2. [이론] 볼륨 마운트의 유형 및 활용
도커에서 데이터를 영구 보존하거나 공유하기 위해 사용하는 `-v` (또는 `--volume`) 옵션의 활용법입니다.

*   **호스트 경로 마운트 (`-v [Host_Path]:[Container_Path]`)**: 호스트의 특정 디렉토리를 컨테이너 디렉토리에 연결합니다. 컨테이너가 삭제되어도 데이터는 호스트 디렉토리에 그대로 유지됩니다.
*   **읽기 전용 마운트 (`:ro`)**: 컨테이너가 호스트의 데이터를 수정하지 못하도록 보호해야 할 때 사용합니다(예: 웹 콘텐츠 배포).
*   **익명 볼륨(자동 마운트)**: 호스트 경로를 지정하지 않고 컨테이너 경로만 지정하면, 도커가 자동으로 UUID 형태의 임시 디렉토리를 생성하여 관리합니다.
*   **컨테이너 간 데이터 공유**: 여러 컨테이너가 동일한 호스트 볼륨을 마운트하면, 한 컨테이너가 생성한 데이터를 다른 컨테이너가 실시간으로 읽거나 처리할 수 있습니다.

## 3. [실습] 스토리지 관리 및 공유 실습

### 예제 1: MySQL 데이터베이스 영구 보존
1.  **실행**: `-v /db_data:/var/lib/mysql` 옵션을 주어 MySQL 컨테이너를 실행합니다.
2.  **검증**: 컨테이너 내부에서 데이터베이스를 생성하면, 호스트의 `/db_data` 폴더에도 해당 데이터 파일이 생성됩니다.
3.  **삭제 및 복구**: 컨테이너를 삭제한 후 다시 동일한 볼륨 옵션으로 실행하면, 이전 데이터가 그대로 보존되어 서비스가 이어지는 것을 확인할 수 있습니다.

### 예제 2: 웹 서버 콘텐츠 관리 (ReadOnly)
1.  **준비**: 호스트에서 미리 `index.html` 파일을 작성합니다.
2.  **실행**: 엔진엑스(Nginx) 컨테이너를 실행할 때 호스트의 웹 데이터 폴더를 `:ro` 옵션으로 마운트합니다.
3.  **확인**: 호스트에서 소스 파일을 수정하면 재시작 없이도 웹페이지에 실시간으로 반영됩니다.

### 예제 3: 컨테이너 간 데이터 공유 (모니터링 시나리오)
1.  **컨테이너 A**: 10초마다 디스크 사용량(`df -h`)을 측정하여 공유 볼륨의 `index.html`에 기록합니다.
2.  **컨테이너 B**: 웹 서버 역할을 하며 동일한 볼륨을 마운트하여 고객에게 실시간 모니터링 페이지를 제공합니다.
3.  **결과**: 한쪽에서 만든 데이터를 다른 쪽에서 서비스하는 협업 구조를 완성합니다.
