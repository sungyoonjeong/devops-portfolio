# 📚 강의별 핵심 요약

## 1강 — 운영체제 개요

### 핵심 개념
```
운영체제 = 컴퓨터 하드웨어 바로 위에서
사용자·응용프로그램에 서비스 제공하는 소프트웨어

역할:
① 자원 관리 (CPU·메모리·디스크·I/O)
② 사용자 인터페이스 제공
③ 프로세스 관리
```

### DevOps 연결
```
컨테이너 = OS 위에서 동작
Docker = Linux OS 기능 (Namespace·Cgroups) 활용
K8s = 여러 OS 위에서 컨테이너 오케스트레이션
```

## 2~3강 — System Structure & Program Execution

### 핵심 개념
```
CPU:
→ 매 클럭마다 메모리에서 인스트럭션 실행
→ Interrupt line 체크

Mode bit:
→ 0: 커널 모드 (OS 실행)
→ 1: 유저 모드 (사용자 프로그램 실행)

시스템 콜 (System Call):
→ 사용자 프로그램이 OS 기능 요청
→ 예) 파일 읽기·네트워크 통신·프로세스 생성

인터럽트:
→ 하드웨어가 CPU에 신호 보내는 것
→ I/O 완료·타이머·예외 상황
```

### DevOps 연결
```
컨테이너 내부 프로그램이 시스템콜 호출
→ Host OS 커널이 처리
→ VM과 달리 Guest OS 없이 Host OS 커널 공유
→ 이게 컨테이너가 VM보다 가벼운 이유
```

## 4강 — 프로세스 1

### 핵심 개념
```
프로세스 = 실행 중인 프로그램

프로세스 상태:
Running  → CPU 실행 중
Ready    → CPU 기다리는 중
Blocked  → I/O 기다리는 중

PCB (Process Control Block):
→ 프로세스 정보 저장하는 자료구조
→ CPU 레지스터·프로그램 카운터·메모리 정보

Context Switch:
→ CPU를 한 프로세스에서 다른 프로세스로 넘기는 것
→ PCB 저장·복원
→ 오버헤드 발생
```

### DevOps 연결
```
컨테이너 = 하나의 프로세스 (or 프로세스 그룹)
K8s Pod = 1개 이상의 컨테이너 (프로세스)
→ Pod 상태 (Running·Pending·Failed) =
   프로세스 상태와 동일한 개념

Context Switch 오버헤드:
→ 컨테이너 수 너무 많으면 성능 저하
→ K8s 리소스 최적화 이유
```

## 5~7강 — 프로세스 2·3

### 핵심 개념
```
프로세스 간 통신 (IPC):
① 메시지 패싱: OS 통해 메시지 전달
② 공유 메모리: 메모리 공간 공유

스레드 (Thread):
→ 프로세스 내 실행 단위
→ 같은 프로세스의 스레드끼리 메모리 공유
→ 멀티스레딩: 응답성↑ 자원 공유 효율↑

멀티프로세스 vs 멀티스레딩:
멀티프로세스: 안전하지만 오버헤드 큼
멀티스레딩: 빠르지만 동기화 문제 발생
```

### DevOps 연결
```
마이크로서비스 = 프로세스 간 통신 (IPC) 구조
→ REST API·gRPC·메시지큐 = 메시지 패싱

K8s 멀티컨테이너 Pod:
→ 같은 Pod 내 컨테이너끼리 localhost 통신
→ 공유 메모리 개념과 유사
```

## 8~9강 — Process Management

### 핵심 개념
```
프로세스 생성:
→ fork(): 부모 프로세스 복제
→ exec(): 새 프로그램으로 덮어씌우기

프로세스 종료:
→ exit(): 자발적 종료
→ abort(): 강제 종료

좀비 프로세스:
→ 자식 종료됐는데 부모가 안 거둬들임

고아 프로세스:
→ 부모가 먼저 종료된 자식
```

### DevOps 연결
```
Docker 컨테이너 시작 = fork() + exec()
컨테이너 종료 = exit()

좀비 프로세스 문제:
→ K8s에서 컨테이너 비정상 종료 시
→ Liveness Probe로 감지·재시작

init 프로세스 (PID 1):
→ 컨테이너 내 첫 번째 프로세스
→ 좀비 프로세스 처리 담당
→ Dockerfile CMD·ENTRYPOINT가 PID 1
```

## 10~11강 — CPU 스케줄링

### 핵심 개념
```
스케줄링 알고리즘:
① FCFS: 먼저 온 순서대로
② SJF: 짧은 작업 먼저
③ RR (Round Robin): 시간 할당량씩 돌아가며
④ Priority: 우선순위 높은 것 먼저

선점형 vs 비선점형:
선점형: CPU 뺏을 수 있음
비선점형: 자발적으로 반납할 때까지 기다림
```

### DevOps 연결
```
K8s 리소스 관리:
→ CPU requests·limits = 스케줄링 기반
→ QoS Class (Guaranteed·Burstable·BestEffort)
   = 우선순위 스케줄링

Pod 우선순위 (PriorityClass):
→ 중요한 Pod 먼저 스케줄링
→ 리소스 부족 시 낮은 우선순위 Pod Eviction
```

## 12~15강 — Process Synchronization

### 핵심 개념
```
Race Condition:
→ 여러 프로세스가 동시에 같은 자원 접근
→ 결과가 접근 순서에 따라 달라짐

Critical Section:
→ 동시에 하나의 프로세스만 접근해야 하는 코드

해결 방법:
① Mutex Lock: 하나씩만 접근
② Semaphore: N개까지 접근 허용
③ Monitor: 고수준 동기화 도구

Deadlock:
→ 두 프로세스가 서로 상대방 자원 기다림
→ 영원히 진행 불가
```

### DevOps 연결
```
분산 시스템 동기화:
→ 여러 컨테이너가 같은 DB 동시 접근
→ Redis 분산 락 = Mutex 개념

K8s Deadlock:
→ Pod A가 Pod B 기다리고
   Pod B가 Pod A 기다리는 상황
→ Liveness Probe로 감지·해결

마이크로서비스 Race Condition:
→ 여러 서비스가 동시에 DB 업데이트
→ 트랜잭션·낙관적 락으로 해결
```

## 16~17강 — Deadlock

### 핵심 개념
```
Deadlock 발생 4조건 (전부 만족해야 발생):
① 상호 배제: 자원을 혼자만 사용
② 비선점: 자원 강제로 뺏을 수 없음
③ 점유 대기: 자원 가진 채로 다른 자원 대기
④ 순환 대기: 프로세스들이 원형으로 대기

해결 방법:
① Prevention: 4조건 중 하나 제거
② Avoidance: 안전한 상태만 허용 (Banker's Algorithm)
③ Detection & Recovery: 탐지 후 복구
④ Ignorance: 그냥 무시 (현실적 선택)
```

### DevOps 연결
```
K8s 리소스 Deadlock:
→ Pod A: CPU 80% 점유·메모리 대기
   Pod B: 메모리 80% 점유·CPU 대기
→ ResourceQuota로 Prevention

마이크로서비스 Deadlock:
→ 서비스 A→B→A 순환 호출
→ Timeout 설정으로 Detection & Recovery
→ Circuit Breaker 패턴 적용
```

## 18~21강 — Memory Management

### 핵심 개념
```
물리 메모리 vs 논리 메모리:
→ 프로그램은 논리 주소 사용
→ OS가 물리 주소로 변환 (MMU)

페이징 (Paging):
→ 메모리를 동일한 크기 페이지로 분할
→ 단편화 없음
→ Page Table로 주소 변환

세그멘테이션:
→ 의미 단위로 분할 (코드·데이터·스택)
→ 외부 단편화 발생
```

### DevOps 연결
```
컨테이너 메모리 관리:
→ K8s memory requests·limits = 물리 메모리 관리
→ OOMKilled: 메모리 초과 시 컨테이너 강제 종료

Docker 메모리 제한:
→ --memory 옵션 = 페이징 기반 메모리 제한
→ Cgroups가 실제 메모리 사용량 제한

Swap:
→ 메모리 부족 시 디스크 사용
→ K8s에서 Swap 비활성화 권장
   (성능 예측 불가능해짐)
```

## 22~23강 — Virtual Memory

### 핵심 개념
```
Demand Paging:
→ 실제 필요할 때만 메모리에 올림
→ Page Fault: 필요한 페이지가 메모리에 없음
→ Page Fault 발생 → 디스크에서 로드

교체 알고리즘:
① FIFO: 먼저 올라온 페이지 교체
② LRU: 가장 오래 안 쓴 페이지 교체
③ Optimal: 가장 오래 안 쓸 페이지 교체

Thrashing:
→ Page Fault 너무 많이 발생
→ CPU 대부분 페이지 교체에 사용
→ 성능 급격히 저하
```

### DevOps 연결
```
컨테이너 Thrashing:
→ 메모리 limits 너무 낮게 설정 시
→ 컨테이너가 계속 OOMKilled 반복
→ Prometheus로 메모리 사용량 모니터링
→ limits 적절히 설정

K8s VPA (Vertical Pod Autoscaler):
→ 자동으로 메모리 requests·limits 조정
→ Thrashing 방지
```

## 27~28강 — Disk Management & Scheduling

### 핵심 개념
```
디스크 스케줄링:

# 🎯 기술면접 핵심 Q&A 30개

## 프로세스·스레드

### Q1. 프로세스와 스레드 차이는?
```
프로세스:
→ 독립적인 메모리 공간 (코드·데이터·힙·스택)
→ 다른 프로세스와 메모리 공유 안 함
→ 생성·전환 비용 큼
→ 하나 죽어도 다른 프로세스 영향 없음

스레드:
→ 프로세스 내 실행 단위
→ 같은 프로세스의 스레드끼리 메모리 공유 (힙·코드·데이터)
→ 스택은 각자 별도
→ 생성·전환 비용 작음
→ 하나가 문제 생기면 전체 프로세스 영향

DevOps 연결:
→ 컨테이너 = 프로세스 격리
→ 멀티스레드 앱을 컨테이너화할 때
   메모리 limits 설정 중요
```

### Q2. Context Switch란?
```
정의:
→ CPU를 한 프로세스에서 다른 프로세스로 넘기는 과정
→ 현재 프로세스 상태(PCB)를 저장하고
   다음 프로세스 상태를 복원

오버헤드:
→ 저장·복원하는 동안 CPU가 유휴 상태
→ 컨테이너 수가 너무 많으면 Context Switch 빈번
→ K8s 노드당 적절한 Pod 수 유지 필요

DevOps 연결:
→ K8s CPU throttling = Context Switch 과다
→ CPU requests 적절히 설정하면 Context Switch 최소화
```

### Q3. 멀티프로세스 vs 멀티스레딩 장단점?
```
멀티프로세스:
장점: 하나가 죽어도 다른 프로세스 영향 없음 (안정성↑)
단점: 메모리 많이 사용 / 프로세스 간 통신 비용 큼

멀티스레딩:
장점: 메모리 공유로 효율적 / 응답성↑ / 생성 비용 작음
단점: 동기화 문제 (Race Condition) / 하나 죽으면 전체 영향

DevOps 연결:
→ 마이크로서비스 = 멀티프로세스 개념
  (서비스 하나 죽어도 다른 서비스 영향 없음)
→ 단일 컨테이너 내부 = 멀티스레딩 사용 가능
```

### Q4. 좀비 프로세스·고아 프로세스란?
```
좀비 프로세스:
→ 자식 프로세스가 종료됐는데
   부모가 wait() 안 해서 PCB가 남아있는 상태
→ PID 낭비

고아 프로세스:
→ 부모 프로세스가 먼저 종료된 자식
→ init 프로세스(PID 1)가 입양해서 처리

DevOps 연결:
→ 컨테이너 PID 1이 자식 프로세스 관리 못하면 좀비 발생
→ tini 같은 init 프로세스 사용 권장
→ K8s Liveness Probe로 좀비 프로세스 감지·재시작
```

## 동기화

### Q5. Race Condition이란?
```
정의:
→ 두 개 이상의 프로세스/스레드가
   동시에 같은 자원에 접근할 때
   실행 순서에 따라 결과가 달라지는 현상

예시:
→ 계좌 잔액 1000원
→ 스레드A: 500원 출금 시작
→ 스레드B: 500원 출금 시작
→ 동시 접근 시 잔액이 0원 or -500원 될 수 있음

해결:
→ Critical Section을 Mutex/Semaphore로 보호

DevOps 연결:
→ 여러 컨테이너가 동시에 같은 DB 레코드 수정
→ Redis 분산 락으로 해결
→ K8s ConfigMap 동시 수정 = Race Condition 가능성
```

### Q6. Mutex와 Semaphore 차이는?
```
Mutex (뮤텍스):
→ 상호 배제 (Mutual Exclusion)
→ 락을 획득한 스레드만 해제 가능
→ 한 번에 하나의 스레드만 접근
→ 이진 세마포어와 유사

Semaphore (세마포어):
→ 카운터 기반
→ N개까지 동시 접근 허용
→ 락 획득한 스레드 외에도 해제 가능
→ 생산자-소비자 패턴에 활용

DevOps 연결:
→ DB Connection Pool = Semaphore 개념
   (최대 N개 연결만 허용)
→ Redis SETNX = 분산 Mutex
→ K8s Rate Limiting = Semaphore 개념
```

### Q7. Deadlock 발생 4조건은?
```
4가지 모두 만족해야 Deadlock 발생:

① 상호 배제 (Mutual Exclusion):
   자원을 한 번에 하나의 프로세스만 사용

② 비선점 (No Preemption):
   자원을 강제로 뺏을 수 없음
   자발적으로 반납할 때까지 기다려야 함

③ 점유 대기 (Hold and Wait):
   자원을 가진 채로 다른 자원을 기다림

④ 순환 대기 (Circular Wait):
   프로세스들이 원형으로 서로 기다림
   A→B→C→A

DevOps 연결:
→ 마이크로서비스 순환 호출:
   서비스A가 B호출 → B가 C호출 → C가 A호출
   → Timeout + Circuit Breaker로 해결
```

### Q8. Deadlock 해결 방법은?
```
① Prevention (예방):
   4조건 중 하나를 원천 제거
   → 자원 순서 정해서 순환 대기 방지

② Avoidance (회피):
   안전한 상태만 허용
   → Banker's Algorithm

③ Detection & Recovery (탐지·복구):
   Deadlock 허용하되 탐지 후 복구
   → 프로세스 강제 종료 or 자원 강제 회수

④ Ignorance (무시):
   Deadlock 드물다고 가정하고 무시
   → 현실에서 가장 많이 사용 (UNIX·Linux)
   → 사용자가 직접 kill

DevOps 연결:
→ K8s Pod Timeout = Detection & Recovery
→ Circuit Breaker = Prevention
→ Health Check + 자동 재시작 = Recovery
```

## 메모리

### Q9. 가상 메모리란?
```
정의:
→ 물리 메모리보다 큰 프로그램 실행 가능하게 하는 기법
→ 프로그램은 연속된 논리 주소 사용
→ OS가 물리 주소로 변환

장점:
→ 메모리보다 큰 프로그램 실행 가능
→ 여러 프로그램 동시 실행 가능
→ 메모리 보호 (프로세스 간 격리)

DevOps 연결:
→ 컨테이너 메모리 격리 = 가상 메모리 기반
→ K8s memory limits = 가상 메모리 상한 설정
→ OOMKiller = 메모리 초과 시 프로세스 강제 종료
```

### Q10. Page Fault란?
```
정의:
→ CPU가 접근하려는 페이지가
   현재 물리 메모리에 없는 상황

처리 과정:
① CPU가 논리 주소 접근
② 해당 페이지가 메모리에 없음 (Page Fault 발생)
③ OS가 디스크에서 해당 페이지 로드
④ Page Table 업데이트
⑤ CPU 재실행

DevOps 연결:
→ 컨테이너 시작 시 Page Fault 많이 발생
   (처음 실행하는 코드·데이터 로드)
→ K8s Readiness Probe:
   Page Fault 완료 후 트래픽 받도록 설정
→ Warm-up 전략으로 Page Fault 최소화
```

### Q11. Thrashing이란?
```
정의:
→ 프로세스들이 필요한 페이지를 메모리에 올리는 것보다
   Page Fault 처리에 더 많은 시간 소비
→ CPU 이용률 급격히 저하

원인:
→ 메모리에 너무 많은 프로세스
→ 각 프로세스가 충분한 메모리 못 받음

해결:
→ Working Set 모델: 자주 쓰는 페이지 보장
→ 프로세스 수 줄이기

DevOps 연결:
→ K8s memory limits 너무 낮게 설정 시
   OOMKilled 반복 = Thrashing과 유사
→ Prometheus로 메모리 사용 패턴 분석
→ VPA(Vertical Pod Autoscaler)로 자동 조정
```

### Q12. 페이징과 세그멘테이션 차이는?
```
페이징 (Paging):
→ 고정 크기 페이지로 분할
→ 내부 단편화 발생 (페이지 내 빈 공간)
→ 외부 단편화 없음
→ 주소 변환: Page Table

세그멘테이션 (Segmentation):
→ 의미 단위 (코드·데이터·스택)로 분할
→ 가변 크기
→ 외부 단편화 발생
→ 내부 단편화 없음

현실:
→ 대부분 페이징 사용 (Linux 포함)
→ x86은 세그멘테이션+페이징 혼합

DevOps 연결:
→ 컨테이너 메모리 = 페이징 기반
→ K8s memory requests = 최소 보장 페이지 수
```

## 스케줄링

### Q13. CPU 스케줄링 알고리즘 종류는?
```
① FCFS (First Come First Served):
   먼저 온 순서대로 처리
   단점: Convoy Effect (긴 작업 뒤에 짧은 작업 대기)

② SJF (Shortest Job First):
   짧은 작업 먼저
   단점: Starvation (긴 작업 영원히 대기 가능)

③ Round Robin (RR):
   시간 할당량(Quantum)씩 돌아가며 실행
   선점형 / 응답 시간 균등
   현실에서 가장 많이 사용

④ Priority Scheduling:
   우선순위 높은 것 먼저
   단점: Starvation → Aging으로 해결

DevOps 연결:
→ K8s QoS Class = Priority Scheduling
→ Guaranteed > Burstable > BestEffort
→ 리소스 부족 시 BestEffort Pod 먼저 Eviction
```

### Q14. 선점형과 비선점형 차이는?
```
선점형 (Preemptive):
→ 실행 중인 프로세스에서 CPU 강제로 뺏을 수 있음
→ 응답 시간 균등
→ Context Switch 오버헤드 있음
→ 예) Round Robin

비선점형 (Non-preemptive):
→ 프로세스가 자발적으로 CPU 반납할 때까지 기다림
→ Context Switch 적음
→ 응답 시간 불균등
→ 예) FCFS

DevOps 연결:
→ Linux = 선점형 OS
→ K8s Pod Eviction = 선점형 개념
   (우선순위 낮은 Pod에서 리소스 강제 회수)
```

### Q15. RR 스케줄링이란?
```
정의:
→ 각 프로세스에 동일한 시간 할당량(Quantum) 부여
→ Quantum 소진 시 다음 프로세스로 넘어감
→ 현대 OS에서 가장 많이 사용

Quantum 크기:
→ 너무 크면: FCFS와 유사해짐
→ 너무 작으면: Context Switch 오버헤드 증가
→ 보통 10~100ms

DevOps 연결:
→ K8s CPU 시간 분배 = RR 기반
→ CPU requests = 최소 보장 시간 할당량
→ CPU limits = 최대 시간 할당량
```

## DevOps 직접 연결 질문

### Q16. 컨테이너와 VM 차이는?
```
VM (Virtual Machine):
→ 하이퍼바이저 위에서 Guest OS 실행
→ 완전한 OS 포함 → 무거움 (GB 단위)
→ 강한 격리
→ 시작 시간 분 단위

컨테이너:
→ Host OS 커널 공유
→ OS 불포함 → 가벼움 (MB 단위)
→ Namespace로 격리
→ 시작 시간 초 단위

핵심 차이:
→ VM: 하드웨어 가상화
→ 컨테이너: OS 수준 가상화
```

### Q17. Docker가 가벼운 이유는?
```
① Host OS 커널 공유:
   Guest OS 없음 → 오버헤드 없음

② Namespace로 격리:
   PID·NET·MNT·UTS·IPC·User
   실제 격리되지만 커널은 공유

③ Cgroups로 자원 제한:
   CPU·메모리·I/O 제한
   실제 물리 자원 직접 사용

④ Union File System (Overlay2):
   레이어 구조로 이미지 공유
   같은 베이스 이미지 중복 저장 안 함
```

### Q18. Namespace와 Cgroups란?
```
Namespace (격리):
→ 프로세스가 보는 시스템 뷰를 격리
→ PID Namespace: 컨테이너 내 PID 1부터 시작
→ NET Namespace: 컨테이너 전용 네트워크 인터페이스
→ MNT Namespace: 컨테이너 전용 파일시스템
→ UTS Namespace: 컨테이너 전용 hostname
→ IPC Namespace: 컨테이너 전용 IPC

Cgroups (자원 제한):
→ 프로세스 그룹의 자원 사용량 제한·측정
→ CPU: 사용량 제한
→ Memory: 메모리 사용량 제한
→ I/O: 디스크 I/O 제한
→ Network: 네트워크 대역폭 제한

Docker = Namespace + Cgroups 조합
K8s limits = Cgroups 설정값
```

### Q19. K8s에서 OOMKilled란?
```
정의:
→ Out Of Memory Killed
→ 컨테이너가 memory limits 초과 시
   Linux OOM Killer가 프로세스 강제 종료

발생 원인:
→ memory limits 너무 낮게 설정
→ 메모리 누수 (Memory Leak)
→ 예상보다 많은 트래픽

해결 방법:
→ Prometheus로 메모리 사용량 모니터링
→ limits 값 적절히 상향 조정
→ VPA로 자동 조정
→ 코드의 메모리 누수 수정
```

### Q20. PID 1 문제란?
```
문제:
→ 컨테이너 내 첫 번째 프로세스 = PID 1
→ PID 1은 시그널 처리·좀비 프로세스 수거 담당
→ 일반 앱(Python·Node.js 등)은 이 역할 제대로 못함

증상:
→ docker stop 시 즉시 종료 안 됨 (SIGTERM 무시)
→ 좀비 프로세스 누적

해결:
→ tini 사용: 경량 init 프로세스
   ENTRYPOINT ["/tini", "--", "앱실행명령"]
→ dumb-init 사용
→ K8s terminationGracePeriodSeconds 설정
```

### Q21. K8s 리소스 requests·limits 차이는?
```
requests (요청):
→ 컨테이너가 보장받아야 하는 최소 자원
→ 스케줄링 기준 (이만큼 여유 있는 노드에만 배치)
→ 반드시 보장됨

limits (제한):
→ 컨테이너가 사용할 수 있는 최대 자원
→ CPU: 초과 시 throttling (느려짐)
→ Memory: 초과 시 OOMKilled (강제 종료)

Best Practice:
→ requests = 평균 사용량
→ limits = 최대 사용량 * 1.5~2배
→ requests = limits: Guaranteed QoS (가장 안정적)
```

### Q22. Liveness Probe·Readiness Probe란?
```
Liveness Probe (생존 확인):
→ 컨테이너가 살아있는지 확인
→ 실패 시: 컨테이너 재시작
→ 용도: Deadlock·무한루프 감지

Readiness Probe (준비 확인):
→ 컨테이너가 트래픽 받을 준비됐는지 확인
→ 실패 시: Service 엔드포인트에서 제거 (트래픽 차단)
→ 용도: 초기화 완료 확인·일시적 과부하 대응

Startup Probe (시작 확인):
→ 컨테이너 시작 완료 확인
→ 시작이 느린 앱에 사용
→ Startup Probe 성공 후 Liveness·Readiness 시작
```

### Q23. K8s에서 Deadlock 어떻게 처리?
```
발생 상황:
→ 마이크로서비스 A→B→C→A 순환 호출
→ 각 서비스가 응답 기다리며 블록

해결 방법:

① Timeout 설정:
   각 서비스 호출에 타임아웃 설정
   일정 시간 내 응답 없으면 에러 반환

② Circuit Breaker:
   특정 서비스 실패율 높아지면
   자동으로 호출 차단 (Fallback 응답)
   Istio·Resilience4j 활용

③ Liveness Probe:
   Deadlock 상태 감지 → 자동 재시작

④ 비동기 통신:
   동기 호출 대신 메시지 큐 사용
   (Kafka·RabbitMQ)
```

### Q24. Docker Overlay2란?
```
정의:
→ Docker의 기본 스토리지 드라이버
→ Union File System 구현체

레이어 구조:
→ 이미지 = 읽기 전용 레이어 여러 개 쌓임
→ 컨테이너 시작 시 맨 위에 읽기/쓰기 레이어 추가

장점:
→ 레이어 공유: 같은 베이스 이미지 사용 시
   레이어 중복 저장 안 함 → 디스크 절약
→ 빠른 빌드: 변경된 레이어만 새로 빌드
→ Copy-on-Write: 파일 수정 시에만 복사

실무:
→ docker images -a: 모든 레이어 확인
→ docker history 이미지명: 레이어 히스토리
→ 레이어 수 줄이면 이미지 크기 최적화
```

### Q25. PV·PVC·StorageClass 차이는?
```
PV (Persistent Volume):
→ 실제 스토리지 자원
→ 관리자가 미리 생성하거나 동적 프로비저닝
→ 클러스터 레벨 자원

PVC (Persistent Volume Claim):
→ Pod가 스토리지를 요청하는 것
→ 원하는 용량·접근 모드 명시
→ 네임스페이스 레벨 자원

StorageClass:
→ 동적 프로비저닝 정의
→ 어떤 종류의 스토리지를 어떻게 생성할지
→ PVC 생성 시 자동으로 PV 생성

관계:
Pod → PVC 요청 → StorageClass → PV 자동 생성
```

### Q26. 컨테이너 non-root 실행 이유는?
```
이유:
→ 컨테이너 내 root = Host의 root와 동일한 권한 가질 수 있음
→ 컨테이너 탈출(Container Escape) 공격 시
   Host 전체 장악 가능

보안 위협:
→ 공격자가 컨테이너 내 앱 취약점 공격
→ root 권한으로 Host 파일시스템 접근
→ 다른 컨테이너 공격

해결:
# Dockerfile
RUN useradd -m appuser
USER appuser

K8s SecurityContext:
securityContext:
  runAsNonRoot: true
  runAsUser: 1000
```

### Q27. K8s QoS Class란?
```
3가지 등급:

① Guaranteed (최우선):
   모든 컨테이너 requests = limits
   → 리소스 부족해도 마지막까지 보호
   → 중요한 프로덕션 워크로드에 사용

② Burstable (중간):
   requests < limits
   → 여유 있을 때 더 사용 가능
   → Guaranteed 다음으로 보호

③ BestEffort (최하위):
   requests·limits 미설정
   → 여유 있을 때만 실행
   → 리소스 부족 시 가장 먼저 Eviction

실무:
→ 중요 서비스: Guaranteed
→ 일반 서비스: Burstable
→ 배치 작업: BestEffort
```

### Q28. CPU throttling이란?
```
정의:
→ 컨테이너가 CPU limits 초과 시
   Linux cgroups가 CPU 사용량 강제로 줄임
→ 프로세스가 느려지지만 종료되지는 않음

발생 원인:
→ CPU limits 너무 낮게 설정
→ 갑작스러운 트래픽 증가

확인 방법:
→ kubectl top pod
→ Prometheus: container_cpu_cfs_throttled_seconds_total
→ Grafana 대시보드에서 CPU throttling 비율 확인

해결:
→ limits 상향 조정
→ HPA로 Pod 수평 확장
→ 코드 최적화
```

### Q29. 컨테이너 네트워크 Namespace란?
```
정의:
→ 각 컨테이너는 독립적인 네트워크 스택 보유
→ 전용 네트워크 인터페이스·IP·라우팅 테이블

K8s 네트워크 모델:
→ 같은 Pod 내 컨테이너: localhost 통신 (네트워크 Namespace 공유)
→ 다른 Pod 간: Pod IP로 통신
→ 서비스 간: Service IP (ClusterIP)로 통신

CNI (Container Network Interface):
→ 컨테이너 네트워크 설정 플러그인
→ 예) Calico·Flannel·Weave·Cilium
→ Pod 간 통신 규칙 정의
→ Network Policy 적용
```

### Q30. Cgroups로 뭘 제어하나?
```
제어 가능한 자원:

① CPU:
   cpu.shares: 상대적 CPU 가중치
   cpu.cfs_quota_us: CPU 사용 한도
   → K8s CPU limits 구현에 사용

② Memory:
   memory.limit_in_bytes: 메모리 상한
   초과 시 OOM Killer 발동
   → K8s memory limits 구현에 사용

③ Block I/O:
   blkio.weight: 디스크 I/O 우선순위
   읽기/쓰기 속도 제한

   네트워크 대역폭 제한
   (tc 명령어와 연동)

⑤ 프로세스 수:
   pids.max: 최대 프로세스 수 제한
   Fork Bomb 방지
