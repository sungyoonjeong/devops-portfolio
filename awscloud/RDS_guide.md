# [쉽게 설명하는 AWS 기초 강좌 25] Amazon RDS 완전 가이드

## 1. RDS(Relational Database Service)란?

### 개념
* **정의:** AWS에서 제공하는 **완전 관리형 관계형 데이터베이스 서비스**로, 하드웨어 프로비저닝·DB 설치·패치·백업 등 인프라 관리를 AWS가 자동으로 처리합니다.
* **핵심 가치:** 사용자는 데이터·스키마·쿼리에만 집중하고, 나머지는 AWS가 담당합니다.

### EC2에 직접 DB 설치 vs RDS
| 항목 | EC2 + DB 직접 설치 | RDS |
| :--- | :--- | :--- |
| **OS 패치** | 직접 | AWS 자동 |
| **DB 설치·업데이트** | 직접 | AWS 자동 |
| **백업** | 직접 | AWS 자동 |
| **고가용성(HA)** | 직접 구성 | Multi-AZ 옵션 |
| **비용** | 저렴 | 상대적으로 비쌈 |
| **커스터마이징** | 자유로움 | 제한 있음 |

### CCNP 연결
```
온프레미스 DB 서버 = 자사 데이터센터에 MySQL 직접 설치
RDS = IDC 임대 후 DB 관리까지 대행사에 맡기는 것
```

## 3. RDS 기본 구성 요소

### DB 인스턴스 (DB Instance)
```
RDS의 기본 단위 = 실행 중인 DB 서버
사양(인스턴스 클래스)에 따라 CPU·메모리 결정

db.t3.micro   : 2 vCPU, 1GB RAM (프리티어)
db.r5.large   : 2 vCPU, 16GB RAM (메모리 최적화)
db.m5.xlarge  : 4 vCPU, 16GB RAM (범용)
```

### DB 서브넷 그룹 (DB Subnet Group)
```
RDS가 배치될 서브넷 목록
최소 2개 AZ의 서브넷 필요 (Multi-AZ 대비)

반드시 프라이빗 서브넷에 배치:
  DB는 인터넷 직접 노출 금지
  웹 서버 보안 그룹에서만 접근 허용
```

### 파라미터 그룹 (Parameter Group)
```
DB 엔진 설정값 집합
예: max_connections, innodb_buffer_pool_size
기본값 사용 → 필요 시 커스텀 파라미터 그룹 생성
```

### 엔드포인트 (Endpoint)
```
DB에 접속하는 주소 (도메인:포트)
예: mydb.c9akciq32.ap-northeast-2.rds.amazonaws.com:3306

IP가 아닌 도메인 사용:
  Failover 시 자동으로 새 DB를 가리킴
  → 앱 코드 변경 불필요
```

## 4. Multi-AZ (다중 가용영역) — 고가용성

### 개념
* **목적:** 고가용성(HA, High Availability) — "장애가 나도 서비스는 계속 된다"
* **구조:**
  * 주 DB(Primary) → 가용영역 A
  * 대기 DB(Standby) → 가용영역 B (다른 AZ)
  * 동기식 복제(Synchronous) — 주 DB에 쓰면 즉시 대기 DB도 동일하게 복제

### 동작 방식
```
[정상 상태]
앱 → 엔드포인트 → 주 DB (AZ-A)
                  ↕ 동기 복제
                  대기 DB (AZ-B) ← 접근 불가 (대기만)

[장애 발생]
주 DB 장애 감지 (보통 1~2분)
→ AWS 자동 Failover
→ 대기 DB를 주 DB로 승격
→ 엔드포인트가 새 주 DB를 자동으로 가리킴
→ 앱은 같은 엔드포인트로 재접속 → 정상 동작

앱 코드 변경 없음!
```

### 자동 Failover 발생 조건
```
주 DB 인스턴스 장애
가용영역 전체 장애
DB 인스턴스 유형 변경
OS 패치 작업
스토리지 장애
네트워크 문제
```

### 핵심 오해 주의
```
❌ 틀린 생각: "Multi-AZ의 대기 DB로 읽기 가능"
✅ 맞는 내용: 대기 DB는 오직 Failover 대기용
              읽기 분산은 Read Replica가 담당

Multi-AZ = 고가용성(HA) 목적
Read Replica = 읽기 성능 향상 목적
```

## 5. Read Replica (읽기 복제본) — 성능 향상

### 개념
* **목적:** 읽기 성능 확장(Scale-out) — "읽기 트래픽을 분산한다"
* **구조:**
  * 주 DB(Primary) → 읽기·쓰기
  * 읽기 복제본(Read Replica) → 읽기 전용 (최대 5~15개)
  * 비동기식 복제(Asynchronous) — 약간의 지연(Replication Lag) 있음

### 동작 방식
```
[쓰기 요청] 앱 → 주 DB
                  ↓ 비동기 복제
[읽기 요청] 앱 → 읽기 복제본 1
            앱 → 읽기 복제본 2
            앱 → 읽기 복제본 3

예: 쇼핑몰
  상품 조회(읽기 90%) → 읽기 복제본으로 분산
  주문·결제(쓰기 10%) → 주 DB 처리
  → 주 DB 부하 90% 감소
```

### 특징
```
비동기 복제: 약간의 시간 지연 발생 가능 (수백ms)
읽기 전용:   복제본에 쓰기 시도하면 에러
별도 엔드포인트: 복제본마다 다른 접속 주소
승격 가능:   복제본을 독립 주 DB로 승격 가능
교차 리전:   다른 리전에도 생성 가능 (재해 복구)

최대 개수:
  RDS: 5개
  Aurora: 15개
```

## 6. Multi-AZ vs Read Replica 완벽 비교

| 항목 | Multi-AZ | Read Replica |
| :--- | :--- | :--- |
| **목적** | 고가용성(HA) | 읽기 성능 확장 |
| **복제 방식** | **동기식** | **비동기식** |
| **대기 DB 읽기** | ❌ 불가 | ✅ 가능 |
| **쓰기 가능** | 주 DB만 | 주 DB만 |
| **자동 Failover** | ✅ 있음 | ❌ 없음 |
| **엔드포인트** | Failover 후 동일 유지 | 복제본별 별도 주소 |
| **위치** | 다른 AZ (필수) | 같은·다른 AZ·리전 |
| **개수** | 1개 대기 DB | 최대 5~15개 |
| **데이터 지연** | 없음 (동기) | 있음 (비동기) |

### SAA 시험 단골 패턴
```
"DB 장애 시 자동 복구가 필요하다" → Multi-AZ
"읽기 트래픽을 분산하고 싶다" → Read Replica
"글로벌 서비스에서 재해 복구가 필요하다" → 교차 리전 Read Replica
"고가용성 + 읽기 성능 모두 원한다" → Multi-AZ + Read Replica 동시 사용
```

## 7. RDS 스토리지 (EBS 기반)

### 스토리지 유형
```
범용 SSD (gp2, gp3):
  일반 DB 워크로드
  gp3 = gp2보다 비용 저렴, 성능 커스텀

프로비저닝된 IOPS (io1, io2):
  높은 I/O 성능 필요한 대규모 DB
  IOPS를 사용자가 직접 지정

마그네틱 (Standard):
  레거시, 현재 거의 사용 안 함
```

### 스토리지 자동 확장 (Storage Auto Scaling)
```
활성화 시:
  스토리지 사용량이 임계값 초과 → 자동 확장
  최대 한도(Maximum Storage Threshold) 설정 가능
  → DB 다운 없이 온라인으로 확장
```

## 8. RDS 백업·복구

### 자동 백업 (Automated Backup)
```
매일 자동 백업 (백업 윈도우 지정 가능)
+ 트랜잭션 로그 5분 단위 저장

보관 기간: 1~35일 (기본 7일)
0으로 설정 시 자동 백업 비활성화

특정 시점 복구(PITR):
  5분 전 상태로도 복구 가능
  새 DB 인스턴스로 생성됨 (기존 덮어쓰기 ❌)
```

### 수동 스냅샷 (Manual Snapshot)
```
사용자가 직접 생성
명시적으로 삭제할 때까지 보관 (기간 제한 없음)
다른 리전·계정으로 복사 가능

용도: 중요 배포 전 백업, 장기 보관
```

### 복구 방법
```
스냅샷 → 새 RDS 인스턴스 생성
PITR → 특정 시점 새 인스턴스 생성

주의: 기존 인스턴스에 덮어쓰기 불가
     새 인스턴스 → 기존과 교체하는 방식
```

## 9. RDS 보안

### 네트워크 보안
```
프라이빗 서브넷 배치 (권장):
  DB는 인터넷 직접 노출 금지

보안 그룹:
  웹 서버 보안 그룹에서만 DB 포트 허용
  예: db-sg 인바운드 3306 → web-sg만 허용
  
  인터넷(0.0.0.0/0)에서 직접 접근 절대 금지
```

### 암호화
```
저장 데이터 암호화 (At-Rest):
  KMS 키 사용
  인스턴스·스냅샷·Read Replica 모두 암호화

전송 중 암호화 (In-Transit):
  SSL/TLS 사용

주의: 암호화는 인스턴스 생성 시에만 설정 가능
      이후 변경 불가 (새 인스턴스 생성 후 마이그레이션)
```

### IAM 인증
```
DB 비밀번호 대신 IAM 토큰으로 접속
자동 만료 (15분) → 보안 강화
MySQL·PostgreSQL 지원
```

## 10. Amazon Aurora — AWS 고성능 DB

### 개념
```
AWS가 클라우드용으로 재설계한 관계형 DB
MySQL·PostgreSQL 완전 호환
→ 기존 앱 코드 변경 없이 마이그레이션 가능
```

### RDS vs Aurora
| 항목 | RDS MySQL | Aurora MySQL |
| :--- | :--- | :--- |
| **성능** | 기준 | **5배** 빠름 |
| **스토리지** | 최대 64TB | **128TB** 자동 확장 |
| **Read Replica** | 최대 5개 | **최대 15개** |
| **복제 속도** | 비동기 | **밀리초 수준** |
| **복사본** | 1개 AZ | **6개 복사본, 3개 AZ** |
| **비용** | 기준 | 20% 비쌈 |
| **Failover** | 1~2분 | **30초 이내** |

### Aurora 스토리지 구조
```
6개 복사본을 3개 AZ에 자동 분산:
  AZ-A: 2개 복사본
  AZ-B: 2개 복사본
  AZ-C: 2개 복사본

4개 복사본 정상이면 쓰기 가능
3개 복사본 정상이면 읽기 가능
→ 2개 AZ 동시 장애도 견딤
```

### Aurora Serverless
```
트래픽에 따라 자동 확장·축소
사용 안 하면 0으로 축소 → 비용 최소화
예측 불가능한 워크로드에 적합
```

## 11. DynamoDB (NoSQL)

### 개념
```
완전 관리형 NoSQL 데이터베이스
키-값(Key-Value) + 문서(Document) 모델
서버리스 (인프라 관리 불필요)
밀리초 미만 응답 속도
자동 확장 (Unlimited)
```

### RDS vs DynamoDB
| 항목 | RDS | DynamoDB |
| :--- | :--- | :--- |
| **유형** | 관계형(SQL) | NoSQL |
| **스키마** | 고정 | 유연 |
| **조인** | ✅ 가능 | ❌ 없음 |
| **확장** | 수직 확장 주로 | 수평 확장 자동 |
| **응답속도** | ms~s | ms 미만 |
| **적합 케이스** | 복잡한 쿼리·트랜잭션 | 대규모·초고속 |

### 언제 쓰나
```
DynamoDB 적합:
  게임 플레이어 데이터 (수천만 동시)
  IoT 센서 데이터 수집
  실시간 순위표(리더보드)
  세션 저장
  장바구니·사용자 프로필

RDS 적합:
  주문·결제 시스템 (복잡한 트랜잭션)
  재무·ERP 시스템
  복잡한 관계형 데이터
```

## 12. ElastiCache (인메모리 캐시)

### 개념
```
인메모리 캐시 서비스
자주 조회하는 데이터를 RAM에 저장
→ DB 부하 감소, 응답 속도 향상 (밀리초 미만)
```

### 엔진 비교
| 항목 | Redis | Memcached |
| :--- | :--- | :--- |
| **데이터 구조** | 다양 (String·List·Set 등) | 단순 문자열 |
| **영속성** | ✅ 지원 | ❌ 없음 |
| **복제·고가용성** | ✅ 지원 | ❌ 없음 |
| **클러스터** | ✅ 지원 | ✅ 지원 |
| **멀티스레드** | ❌ | ✅ |
| **적합 케이스** | 세션·순위표·메시지큐 | 단순 캐싱 |

### 캐시 패턴
```
캐시 어사이드 (Cache-Aside):
  앱 → Redis 조회 → 없으면 DB 조회 → Redis에 저장
  가장 많이 쓰는 패턴

라이트 스루 (Write-Through):
  DB 쓸 때 동시에 Redis도 업데이트
  데이터 항상 최신, 쓰기 부하 증가

세션 저장:
  사용자 로그인 세션을 Redis에 저장
  여러 서버가 공유 가능 (Scale-out에 필수)
```

### DevOps 활용
```
웹 앱 아키텍처:
  앱 서버 → ElastiCache 확인 → RDS

K8s 앱:
  Pod들이 외부 ElastiCache 공유
  → 세션 공유 문제 해결
```

## 14. 전형적인 3-Tier 아키텍처

```
인터넷
  ↓
ALB (로드밸런서) — 퍼블릭 서브넷
  ↓
EC2 웹 서버 Auto Scaling Group — 퍼블릭 서브넷
  ↓ (보안 그룹으로 제어)
ElastiCache Redis — 프라이빗 서브넷 (캐시 레이어)
  ↓ (캐시 미스 시)
RDS MySQL Multi-AZ — 프라이빗 서브넷 (DB 레이어)
  ↓ (읽기 분산)
Read Replica ×3 — 프라이빗 서브넷

이 구조가 DevOps/SAA 시험에서 가장 자주 나오는 표준 아키텍처
PF1에서도 이 구조 참고해서 설계
```

## 15. SAA·면접 핵심 Q&A

**Q. Multi-AZ와 Read Replica의 차이는?**
```
Multi-AZ는 고가용성이 목적입니다.
다른 AZ에 대기 DB를 두고 동기식으로 복제하며,
주 DB 장애 시 자동 Failover됩니다.
대기 DB는 읽기에 사용할 수 없습니다.

Read Replica는 읽기 성능 확장이 목적입니다.
비동기식 복제로 읽기 트래픽을 분산하며,
자동 Failover는 없습니다.
```

**Q. RDS를 프라이빗 서브넷에 두는 이유는?**
```
DB는 민감한 데이터를 담고 있어 인터넷에
직접 노출되면 안 됩니다.
프라이빗 서브넷에 배치하고 보안 그룹으로
웹 서버에서만 DB 포트 접근을 허용합니다.
```

**Q. ElastiCache는 언제 쓰나요?**
```
자주 조회되는 데이터를 메모리에 캐싱해
DB 부하를 줄이고 응답 속도를 높일 때 씁니다.
세션 저장, 실시간 순위표, 자주 조회하는
DB 쿼리 결과 캐싱 등에 활용합니다.
```

**Q. Aurora가 일반 RDS보다 나은 점은?**
```
MySQL 대비 5배 성능, 128TB까지 자동 확장,
3개 AZ에 6개 복사본으로 내구성이 높습니다.
읽기 복제본도 최대 15개까지 가능하고
Failover가 30초 이내로 빠릅니다.
```

## 16. 핵심 치트시트

| 개념 | 한줄 요약 |
| :--- | :--- |
| RDS | 관리형 관계형 DB (패치·백업 자동) |
| Multi-AZ | 동기 복제, 고가용성, 자동 Failover |
| Read Replica | 비동기 복제, 읽기 분산, 최대 5~15개 |
| PITR | 특정 시점 복구 (5분 전까지) |
| Aurora | AWS 고성능 DB (MySQL 5배, 128TB) |
| DynamoDB | NoSQL, 서버리스, 밀리초 응답 |
| ElastiCache | 인메모리 캐시 (Redis·Memcached) |
| Redshift | 데이터웨어하우스, 빅데이터 분석 |
