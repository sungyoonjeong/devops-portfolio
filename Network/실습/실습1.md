## 실습 1. traceroute — 라우팅 경로 추적

### 명령어
```bash
traceroute 8.8.8.8
```

### 실제 결과
```
traceroute to 8.8.8.8 (8.8.8.8), 30 hops max, 60 byte packets
 1  172.17.144.1 (172.17.144.1)   0.541 ms  0.510 ms  0.433 ms
 2  192.168.100.1 (192.168.100.1)  2.427 ms  2.408 ms  2.389 ms
 3  * * *
 4  * * *
 5  * * *
... (이후 전부 * * *)
```

### 결과 분석
```
hop 1: 172.17.144.1
→ WSL2 가상 네트워크 게이트웨이
→ WSL2는 Windows 호스트와 가상 네트워크로 연결됨
→ 0.5ms = 같은 PC 안이라 매우 빠름

hop 2: 192.168.100.1
→ 실제 공유기(홈 라우터) IP
→ 2.4ms = LAN 구간 지연

hop 3 이후: * * *
→ * = 해당 라우터가 ICMP 응답을 차단함
→ 보안 정책으로 traceroute 응답을 막는 라우터가 많음
→ 패킷 손실이 아니라 응답 거부 (실제 통신은 정상)
→ 구글 8.8.8.8까지 패킷은 정상적으로 도달함
   (ping 실습에서 확인됨)
```

### 이석복 강의 연결
```
강의 핵심 개념:
→ 패킷은 목적지까지 라우터를 hop by hop으로 이동
→ 각 hop = 라우터 1개
→ 각 hop에서 처리 지연 + 전파 지연 발생
→ ms 값 = 해당 라우터까지의 RTT (왕복 지연 시간)

실습에서 확인:
→ 내 PC → WSL2 게이트웨이(0.5ms) → 공유기(2.4ms) → ...
→ 멀어질수록 지연 시간 증가
```


### 결과 분석
```
도메인: google.com
IPv4 주소 4개:
→ 142.251.24.113
→ 142.251.24.100
→ 142.251.24.102
→ 142.251.24.139

IPv6 주소 4개:
→ 2404:6800:400b:c00e::8b 등

왜 IP가 여러 개인가?
→ 구글은 전 세계 수억 명이 접속
→ 서버 1대로는 불가능 → 수천 대의 서버 운영
→ DNS가 여러 IP를 돌아가며 응답 (Round Robin DNS)
→ 사용자마다 다른 서버로 분산 → 로드 밸런싱 효과

DNS 동작 원리:
브라우저 → "google.com의 IP가 뭐야?" → DNS 서버
DNS 서버 → "142.251.24.113이야" → 브라우저
브라우저 → 142.251.24.113에 TCP 연결 → HTTP 요청
```

### 이석복 강의 연결
```
강의 핵심 개념:
→ DNS = Domain Name System
→ 사람이 읽기 쉬운 도메인을 컴퓨터가 쓰는 IP로 변환
→ 포트 53 사용 (UDP → 빠른 응답 필요)
→ 계층적 구조: 루트 DNS → TLD DNS(.com) → 권한 DNS

실습에서 확인:
→ google.com → 142.251.24.113 (IPv4)
→ google.com → 2404:6800:... (IPv6)
→ 여러 IP = CDN·로드밸런싱 구조 직접 확인
```

## 실습 3. curl — HTTP 요청·응답 확인

### 명령어
```bash
curl -v http://www.google.com 2>&1 | head -30
```

### 실제 결과
```
* Trying 142.251.157.119:80...          ← TCP 연결 시도 (IP:포트)
* Connected to www.google.com port 80   ← TCP 연결 성공

> GET / HTTP/1.1                        ← HTTP 요청 라인 (메서드 + 경로 + 버전)
> Host: www.google.com                  ← 요청할 서버 호스트명
> User-Agent: curl/7.81.0              ← 클라이언트 정보 (curl 버전)
> Accept: */*                           ← 모든 형식의 응답 수락
>                                       ← 빈 줄 = 요청 헤더 끝

< HTTP/1.1 200 OK                       ← HTTP 응답 상태 코드 (200 = 성공)
< Content-Type: text/html; charset=ISO-8859-1  ← 응답 파일 형식
< Date: Wed, 27 May 2026 02:01:17 GMT   ← 응답 시각(GMT시간은 한국표준시각 -9시간)
< Expires: -1                           ← 캐시 만료 (즉시 만료)
< Cache-Control: private, max-age=0     ← 캐시 정책
< Server: gws                           ← 구글 웹 서버 이름
< X-XSS-Protection: 0                  ← XSS 보안 헤더
< X-Frame-Options: SAMEORIGIN          ← 클릭재킹 방지
< Set-Cookie: __Secure-STRP=...        ← 쿠키 설정 (세션 관리)
< Accept-Ranges: none
< Vary: Accept-Encoding
< Transfer-Encoding: chunked            ← 청크 단위 전송
<
<!doctype html><html ...>               ← 실제 HTML 본문 시작
```

### 결과 분석
```
요청 흐름:
1. DNS로 www.google.com → 142.251.157.119 변환
2. 142.251.157.119:80으로 TCP 3-Way Handshake
3. HTTP GET / HTTP/1.1 요청 전송
4. 서버가 200 OK + HTML 응답
5. TCP 연결 종료

> 기호 = 클라이언트가 서버에게 보낸 것 (요청)
< 기호 = 서버가 클라이언트에게 보낸 것 (응답)

HTTP 상태 코드:
200 OK      → 요청 성공, 본문 있음
301 Moved   → 영구 리다이렉트 (http → https)
404 Not Found → 파일 없음
500 Internal → 서버 오류

포트 80:
→ HTTP 기본 포트
→ 브라우저는 http:// 입력 시 자동으로 포트 80 사용
→ https:// = 포트 443
```

### Project 1 연결
```
이 실습이 Project 1과 직접 연결됨:

curl이 보내는 것:        내가 만든 서버가 받는 것:
GET / HTTP/1.1      →   read()로 수신
Host: ...           →   파싱해서 경로 추출
                        파일 읽기
HTTP/1.1 200 OK     ←   write()로 전송
Content-Type: ...   ←
[HTML 데이터]       ←

→ curl의 동작이 정확히 myserver_partB.py의 handle_client()와 대응
```

### 이석복 강의 연결
```
강의 핵심 개념:
→ HTTP = HyperText Transfer Protocol (응용 계층)
→ TCP 위에서 동작 (포트 80/443)
→ stateless: 서버는 이전 요청 기억 안 함
→ 요청 메서드: GET(조회) POST(생성) PUT(수정) DELETE(삭제)

실습에서 확인:
→ TCP 연결 → HTTP 요청 → HTTP 응답 → 연결 종료
→ 요청 헤더(Host, User-Agent, Accept)
→ 응답 헤더(Content-Type, Set-Cookie, Cache-Control)
→ 200 OK = 정상 응답 직접 확인
```

## 실습 4. arp — ARP 테이블 확인

### 명령어
```bash
arp -a
```

### 실제 결과
```
? (172.17.144.1) at 00:15:5d:80:ab:cd [ether] on eth0
```

### 결과 분석
```
172.17.144.1  → WSL2 가상 게이트웨이 IP 주소
00:15:5d:80:ab:cd → 해당 게이트웨이의 MAC 주소
[ether]       → 이더넷 방식
eth0          → WSL2 네트워크 인터페이스 이름

ARP 테이블이 1개뿐인 이유:
→ WSL2는 가상 네트워크 환경
→ 실제 LAN이 아니라서 같은 네트워크 장치가 게이트웨이 1개뿐
→ 실제 물리 PC의 arp -a는 공유기·TV·스마트폰 등 다수 표시

MAC 주소 형식:
→ 00:15:5d:80:ab:cd
→ 6바이트 (48비트)
→ 00:15:5d = 제조사 식별자 (Microsoft Hyper-V)
→ 80:ab:cd = 장치 고유 번호
```

### 이석복 강의 연결
```
강의 핵심 개념:
→ ARP = Address Resolution Protocol (링크 계층)
→ IP 주소(3계층) → MAC 주소(2계층) 변환
→ 같은 LAN 안에서 실제 통신할 때 MAC 주소 필요
→ ARP Request: "172.17.144.1 쓰는 사람 MAC 주소 알려줘!"
→ ARP Reply: "나야, 00:15:5d:80:ab:cd"

실습에서 확인:
→ 게이트웨이 172.17.144.1의 MAC 주소가 캐시에 저장됨
→ traceroute에서 hop 1로 나왔던 IP와 동일
→ 패킷이 게이트웨이로 나갈 때 이 MAC 주소 사용
```

```bash
netstat -an | grep LISTEN
```

```
tcp    0   0  0.0.0.0:10.255.254.53   0.0.0.0:*   LISTEN
tcp    0   0  127.0.0.1:45889         0.0.0.0:*   LISTEN

unix   2  [ACC]  STREAM  LISTENING  7211  /run/WSL/1_interop
unix   2  [ACC]  STREAM  LISTENING  7179  /run/WSL/9_interop
unix   2  [ACC]  STREAM  LISTENING  5846  /run/WSL/9_interop
... (이하 unix 소켓 다수)
```

### 결과 분석
```
TCP 소켓 (네트워크 통신):
┌──────────────────────────────────────────────────────────────┐
│ 0.0.0.0:10.255.254.53                                        │
│ → IP: 0.0.0.0 = 모든 인터페이스에서 수신                    │
│ → 포트: 53 = DNS 포트                                        │
│ → WSL2 내부 DNS 서버가 실행 중                               │
│                                                              │
│ 127.0.0.1:45889                                              │
│ → IP: 127.0.0.1 = 루프백 (localhost만 접근 가능)             │
│ → 포트: 45889 = WSL2 내부 서비스                             │
└──────────────────────────────────────────────────────────────┘

Unix 소켓 (같은 PC 내 프로세스 간 통신):
→ /run/WSL/1_interop = WSL2 ↔ Windows 통신 소켓
→ /run/wslg = WSL GUI 관련 소켓
→ /var/run/dbus = 시스템 메시지 버스
→ 네트워크가 아닌 파일 시스템 기반 IPC

내 서버 실행 시 표시되는 것:
→ python3 myserver_partB.py 7777 실행하면
  tcp  0  0  0.0.0.0:7777  0.0.0.0:*  LISTEN 추가됨
```

### 이석복 강의 연결
```
강의 핵심 개념:
→ 포트 번호 = 같은 IP에서 여러 서비스 구분
→ 서버 프로세스는 특정 포트에서 LISTEN
→ 클라이언트는 임시 포트(49152~65535) 사용
→ LISTEN = 연결 대기 상태 (listen() 호출 후)

Well-known 포트:
→ 22:  SSH
→ 53:  DNS  ← 실습에서 확인됨
→ 80:  HTTP
→ 443: HTTPS
→ 7777: 내 웹 서버 (Project 1)

실습에서 확인:
→ 53번 포트 LISTEN = WSL2 DNS 서버 동작 중
→ 서버가 listen() 호출하면 netstat에 LISTEN으로 표시됨
→ Project 1 서버 실행 후 다시 확인하면 7777 추가됨
```

## 실습 6. ping — RTT 측정

### 명령어
```bash
ping -c 4 google.com
```

### 실제 결과
```
PING google.com (142.251.24.139) 56(84) bytes of data.
64 bytes from rj-in-f139.1e100.net (142.251.24.139): icmp_seq=1 ttl=111 time=36.1 ms
64 bytes from rj-in-f139.1e100.net (142.251.24.139): icmp_seq=2 ttl=111 time=36.7 ms
64 bytes from rj-in-f139.1e100.net (142.251.24.139): icmp_seq=3 ttl=111 time=36.3 ms
64 bytes from rj-in-f139.1e100.net (142.251.24.139): icmp_seq=4 ttl=111 time=37.0 ms

--- google.com ping statistics ---
4 packets transmitted, 4 received, 0% packet loss, time 3004ms
rtt min/avg/max/mdev = 36.073/36.527/36.986/0.357 ms
```

### 결과 분석
```
목적지: google.com → 142.251.24.139 (DNS 변환 자동)
패킷 크기: 56(84) bytes
→ ICMP 헤더 28바이트 + 데이터 56바이트 = 84바이트

각 줄 분석:
icmp_seq=1   → 1번째 패킷
ttl=111      → Time To Live (남은 홉 수)
             → 최초 TTL이 보통 128 → 128-111=17홉 경유
time=36.1ms  → RTT (이 패킷의 왕복 지연 시간)

통계:
4 packets transmitted  → 4개 보냄
4 received             → 4개 받음
0% packet loss         → 손실 없음 ✅
rtt min=36.073ms       → 최소 지연
rtt avg=36.527ms       → 평균 지연 ≈ 36.5ms
rtt max=36.986ms       → 최대 지연
mdev=0.357ms           → 지연 편차 (매우 안정적)

traceroute와 비교:
→ traceroute에서 hop 3 이후 * * * 였지만
→ ping은 4/4 성공 → 패킷은 정상 도달
→ 중간 라우터들이 ICMP 응답을 거부했을 뿐
→ 8.8.8.8까지 실제 경로: 약 17홉 소요
```

### 이석복 강의 연결
```
강의 핵심 개념:
→ RTT (Round Trip Time) = 패킷 왕복 시간
→ 전파 지연 = 물리적 거리에 의한 지연
→ 서울 → 구글 서버(싱가포르 추정) = 약 36ms RTT
→ 빛의 속도 2×10^8 m/s 기준
  36ms/2 = 18ms (편도)
  18ms × 2×10^8 m/s = 약 3,600km 거리 추정

패킷 지연 4가지와 연결:
→ 처리 지연: 각 라우터에서 다음 경로 결정
→ 큐잉 지연: 라우터 혼잡 시 대기
→ 전송 지연: 링크에 패킷 밀어넣는 시간
→ 전파 지연: 서울→싱가포르 물리적 거리 → 주요 지연 원인

실습에서 확인:
→ 평균 36.5ms RTT = 18ms 편도 전파 지연
→ 0% 패킷 손실 = 네트워크 정상
→ mdev 0.357ms = 지연 변동 매우 작음 (안정적)
```
