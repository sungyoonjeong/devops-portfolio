# 🌐 [강의 노트] 21강. 멀티미디어 네트워킹 (Multimedia Networking)

이 강의는 멀티미디어 데이터의 디지털화 과정부터 네트워크의 지터(Jitter) 문제를 해결하는 버퍼링 기술, 그리고 현대 스트리밍의 핵심인 DASH와 CDN 기술을 다룹니다.

---

## 1. 멀티미디어 데이터의 디지털화 (Digitalization)

아날로그 신호(음성, 영상 등)를 네트워크로 전송하기 위해서는 디지털 데이터로 변환하는 과정이 필요합니다.

### 1.1 오디오 (Audio)
*   **샘플링(Sampling):** 아날로그 신호를 일정한 주기(Sampling Rate)로 측정하여 디지털 값으로 추출합니다.
    *   전화: 초당 8,000번 샘플링.
    *   CD 음악: 초당 44,100번 샘플링.
*   **양자화(Quantization):** 샘플링된 아날로그 값을 디지털 비트(예: 8비트, 256개 값)로 변환합니다. 비트 수가 많을수록 오차(Quantization Error)가 줄어들고 원음에 가까워집니다.
*   **코딩 레이트(Coding Rate):** 초당 표현되는 비트 수(bps)입니다. 코딩 레이트가 높을수록 음질은 좋지만 전송량이 많아집니다.
    *   MP3: 96, 128, 160 kbps.
    *   CD: 1.411 Mbps.

### 1.2 비디오 (Video)
*   **이미지의 연속:** 비디오는 일정한 간격으로 표시되는 이미지(Frame)들의 나열입니다 (예: 초당 24장).
*   **압축 기술 (Coding):** 중복된 정보를 제거하여 데이터 크기를 줄입니다.
    *   **공간적 압축(Spatial):** 하나의 이미지 내에서 이웃한 비슷한 픽셀 정보를 통합하여 압축합니다.
    *   **시간적 압축(Temporal):** 이전 프레임과 다음 프레임 간의 차이점(변화된 부분)만 기록하여 압축합니다.
*   **전송 방식:**
    *   **CBR (Constant Bit Rate):** 고정된 전송률로 인코딩.
    *   **VBR (Variable Bit Rate):** 영상의 복잡도에 따라 전송률이 변함.

## 2. 멀티미디어 애플리케이션의 종류

1.  **Streaming Stored Audio/Video:** 서버에 저장된 파일을 클라이언트가 재생하는 방식 (예: 유튜브, 넷플릭스).
2.  **Conversational Voice/Video over IP:** 실시간 상호작용이 필요한 대화형 통신 (예: 스카이프).
3.  **Streaming Live Audio/Video:** 실시간으로 발생하는 이벤트를 중계하는 방식 (예: 야구 생중계).

## 3. 스트리밍의 핵심 문제와 해결책

### 3.1 네트워크 지터 (Network Jitter)
서버가 일정한 속도로 데이터를 보내더라도, 네트워크 상황에 따라 패킷이 도착하는 시간 간격이 불규칙해지는 현상을 **지터(Jitter)**라고 합니다.

### 3.2 클라이언트 측 버퍼링 (Client-side Buffering)
지터 문제를 해결하기 위해 클라이언트는 데이터를 받자마자 재생하지 않고, **어느 정도 버퍼를 채운 뒤 재생**을 시작합니다.
*   **효과:** 네트워크가 잠시 느려져도 미리 받아둔 데이터를 재생하여 끊김 현상을 방지합니다.
*   **트레이드오프:** 버퍼를 많이 채울수록 끊김은 줄어들지만, 사용자가 영상을 보기까지 기다려야 하는 초기 지연 시간(Initial Delay)이 길어집니다.

## 4. DASH (Dynamic, Adaptive Streaming over HTTP)

유튜브 같은 현대 스트리밍 서비스가 사용하는 핵심 기술입니다.

*   **동작 원리:**
    1.  서버는 큰 영상 파일을 아주 작은 조각인 **청크(Chunk)**(예: 256KB 단위)로 쪼개어 저장합니다.
    2.  각 청크는 다양한 화질(코딩 레이트) 버전으로 인코딩되어 저장됩니다.
    3.  **매니페스트 파일(Manifest File):** 각 버전의 청크가 어디에 저장되어 있는지 알려주는 URL 테이블입니다.
*   **클라이언트의 선택:** 클라이언트는 자신의 현재 네트워크 속도를 측정하여, **감당 가능한 최고 화질의 청크를 하나씩 요청**합니다.
    *   속도가 잘 나오면 고화질 청크를, 속도가 떨어지면 즉시 낮은 화질의 청크를 요청하여 영상이 멈추는(버퍼링) 것을 방지합니다.

---

## 5. CDN (Content Distribution Network)

수많은 사용자가 동시에 몰릴 때 발생하는 서버 부하와 네트워크 혼잡을 해결하기 위한 인프라 기술입니다.

*   **배경:** 서버 한 곳에서만 데이터를 보내면 전 세계 사용자의 요청을 감당하기 어렵고, 거리가 멀어 지연이 발생합니다.
*   **해결책:** 전 세계 곳곳에 복사본을 저장한 **에지 서버(Edge Server)**들을 배치합니다.
*   **가장 가까운 서버 찾기:**
    1.  클라이언트가 도메인 주소로 IP를 묻는 **DNS 쿼리**를 보냅니다.
    2.  CDN 업체의 DNS 서버는 요청을 보낸 사용자의 **IP 주소를 확인**하여, 지리적으로 혹은 네트워크상으로 **가장 가까운(홉수가 적은) 에지 서버의 IP**를 알려줍니다.
*   **사례 (Netflix):** 넷플릭스는 계정 관리는 아마존 클라우드에서 하지만, 실제 영상 데이터는 전 세계에 흩어진 여러 CDN 업체(Akamai, Limelight 등)를 통해 제공하여 품질을 극대화합니다.

---

### 💡 21강 요약
21강은 멀티미디어가 단순한 데이터 전송을 넘어, **사용자의 시청 경험(Quality of Experience)**을 보장하기 위해 네트워크 기술이 어떻게 진화했는지 설명합니다. **DASH**를 통해 화질을 유동적으로 조절하고, **CDN**을 통해 데이터 전송 경로를 단축하는 것이 현대 스트리밍 서비스의 핵심입니다.
