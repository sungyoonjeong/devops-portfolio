# 🌐 [학습 정리] 22강. 네트워크 보안의 원리와 암호학 기초

## 1. 네트워크 보안의 4대 요구 조건
네트워크 보안이 실현되기 위해서는 다음 네 가지 핵심 개념이 충족되어야 합니다.

1.  **기밀성 (Confidentiality):** 오직 송신자와 수신자만이 메시지 내용을 이해할 수 있어야 합니다. 이를 위해 송신자는 메시지를 **암호화**하고 수신자는 이를 **복호화**합니다.
2.  **인증 (Authentication):** 내가 지금 통신하고 있는 상대방이 정말 내가 생각하는 그 사람이 맞는지 확신할 수 있어야 합니다.
3.  **메시지 무결성 (Message Integrity):** 전송 과정에서 메시지가 제3자에 의해 변경되거나 위조되지 않았음을 보장해야 합니다.
4.  **가용성 및 접근성 (Access and Availability):** 서비스가 공격(예: DDoS)으로부터 보호되어 정당한 사용자가 언제든 서비스를 이용할 수 있어야 합니다.

## 2. 인터넷 프로토콜 스택과 보안의 현실
현재 우리가 사용하는 인터넷 프로토콜(TCP/IP)은 1970년대 초반에 설계되었을 당시, 학자나 연구소 간의 신뢰를 바탕으로 디자인되었기 때문에 **보안 개념이 전혀 녹아있지 않았습니다**. 따라서 새로운 공격이 발생할 때마다 애플리케이션, 전송, 네트워크 계층 등에 보안 메커니즘을 **"덧대기(Patch)"** 형식으로 추가하다 보니 현재의 보안 구조는 다소 복잡하고 지저분한 상태입니다.

## 3. 보안의 주인공과 적: Alice, Bob, Trudy
암호학 세계에서 통신을 시도하는 두 주인공은 보통 **앨리스(Alice)**와 **밥(Bob)**으로 불리며, 이들의 통신을 방해하거나 훔쳐보는 침입자는 **트루디(Trudy)**라고 불립니다.
*   오늘날의 환경에서는 **누구나 트루디가 될 수 있습니다.** 와이어샤크(Wireshark) 같은 툴을 사용하면 누구나 공용 와이파이 환경 등에서 패킷을 캡처하여 다른 사람의 인터넷 활동(IP 헤더, TCP 데이터 등)을 모니터링할 수 있기 때문입니다.

## 4. 암호학의 두 가지 핵심 방식

### 4.1 대칭키 암호화 (Symmetric Key Cryptography)
*   **원리:** 앨리스와 밥이 **동일한 비밀키($K_S$)**를 공유하여 암호화와 복호화에 모두 사용하는 방식입니다.
*   **특징:** 연산 속도가 매우 빠르고 효율적이지만, **"처음 만난 사람끼리 어떻게 비밀키를 안전하게 공유할 것인가?"**라는 고질적인 문제가 있습니다.
*   **대표 알고리즘:** DES(56비트 키 사용)가 유명하며, 보안성을 높이기 위해 키를 3번 적용하는 3DES 방식도 사용됩니다.

### 4.2 공개키 암호화 (Public Key Cryptography)
*   **원리:** 모든 사용자가 **공개키(Public Key)**와 **개인키(Private Key)**라는 한 쌍의 키를 가집니다.
    *   **공개키:** 모든 사람에게 공개되며, 메시지를 **암호화**할 때 사용합니다.
    *   **개인키:** 오직 본인만 알고 있으며, 암호화된 메시지를 **복호화**할 때 사용합니다.
*   **장점:** 사전에 비밀키를 공유할 필요가 없어 혁신적입니다.
*   **RSA 알고리즘:** MIT 교수들의 이름을 딴 이 방식은 거대한 숫자의 소인수분해가 어렵다는 수학적 원리를 이용하며, **"어떤 키를 먼저 적용하든 순서에 관계없이 결과가 동일하다"**는 유용한 특성을 가집니다.
*   **단점:** 수학적 연산이 복잡하여 대칭키 방식보다 훨씬 느립니다. 따라서 실제로는 공개키로 **대칭키를 공유**한 뒤, 본 데이터는 대칭키로 주고받는 혼합 방식을 주로 사용합니다.

## 5. 인증 및 메시지 무결성 기술

### 5.1 인증 (Authentication)
상대방이 진짜인지 확인하기 위해 **넌스(Nonce)**라는 일회용 숫자를 챌린지로 보냅니다. 상대방이 자신의 개인키로 이 숫자를 암호화하여 돌려주면, 그의 공개키로 풀어봄으로써 신원을 확인할 수 있습니다.

### 5.2 메시지 무결성과 디지털 서명 (Digital Signatures)
*   **디지털 서명:** 메시지($m$)를 송신자의 개인키로 암호화한 것입니다. 이는 오직 송신자만이 만들 수 있으므로 무결성을 증명합니다.
*   **메시지 다이제스트 (Hash):** 긴 메시지 전체를 공개키로 암호화하는 것은 비효율적이므로, 메시지를 **해시 함수**에 통과시켜 고유한 "지문"인 **메시지 다이제스트**를 만든 뒤 이에 대해서만 서명합니다.

## 6. 신뢰의 뿌리: 인증서(Certificates)와 인증 기관(CA)
공개키 방식의 치명적인 약점은 **"누군가 가짜 공개키를 주며 속일 수 있다"**는 점입니다. 이를 해결하기 위해 **공인 인증 기관(CA)**이 등장합니다.
*   **인증서:** 특정 공개키가 특정인의 것임을 보증하는 문서입니다. CA의 개인키로 서명되어 있습니다.
*   **신뢰의 체인:** 우리는 브라우저에 하드코딩된 **루트(Root) CA의 공개키**를 무조건적으로 신뢰함으로써 전체 보안 시스템을 가동합니다.

# 🌐 [완벽 통합판] 23강. 네트워크 보안 2: SSL/TLS 및 방화벽

23강은 전송 계층과 애플리케이션 계층 사이에서 보안을 담당하는 **SSL**과 네트워크 경계에서 보안을 책임지는 **방화벽**을 중점적으로 다룹니다.

## 1. SSL (Secure Sockets Layer)과 TLS

### 1.1 필요성 및 정의
*   **TCP의 한계:** TCP 프로토콜은 신뢰성 있는 전송은 보장하지만, **보안(Security) 기능은 전혀 제공하지 않습니다**. 따라서 HTTP와 같은 애플리케이션 메시지가 TCP 소켓을 통해 그대로 내려가면 외부에서 누구나 내용을 엿볼 수 있습니다.
*   **SSL의 등장:** 이를 해결하기 위해 애플리케이션과 TCP 사이에 삽입된 보안 라이브러리가 SSL입니다. 엄밀히 말하면 새로운 계층은 아니며, **애플리케이션 계층에 존재하는 라이브러리**입니다.
*   **용어의 변화:** 현재는 SSL보다 **TLS (Transport Layer Security)**라는 용어가 더 표준으로 사용됩니다.
*   **HTTPS:** 우리가 흔히 쓰는 `https://`는 **HTTP 프로토콜을 SSL을 통해 전송**하는 것을 의미합니다 (HTTPS = HTTP + SSL).

### 1.2 SSL의 동작 과정 (Handshake ~ Closure)
SSL 통신은 크게 4단계로 이루어집니다.

1.  **핸드셰이크 (Handshake):**
    *   먼저 **TCP 커넥션**이 생성되어야 합니다.
    *   클라이언트가 헬로(Hello) 메시지를 보내면, 서버는 자신의 **공개키 인증서**를 보냅니다.
    *   클라이언트는 인증 기관(CA)의 공개키로 이를 검증하여 서버의 신원을 확인하고, 서버의 공개키를 얻습니다.
2.  **키 유도 (Key Derivation):**
    *   클라이언트는 **마스터 시크릿(Master Secret)**을 생성하여 서버의 공개키로 암호화해 보냅니다.
    *   이 마스터 시크릿을 기반으로 **4가지 종류의 키**를 만듭니다.
        *   **Client->Server:** 암호화용 키 / 무결성(MAC)용 키
        *   **Server->Client:** 암호화용 키 / 무결성(MAC)용 키
    *   키를 4개로 나누는 이유는 특정 키가 유출되었을 때의 피해 범위를 최소화하기 위해서입니다.
3.  **데이터 전송 (Data Transfer):**
    *   데이터를 일정한 크기의 **레코드(Record)** 단위로 쪼개어 보냅니다.
    *   각 레코드에는 **데이터 길이, 타입, 데이터, 그리고 MAC**이 포함됩니다.
    *   **MAC (Message Authentication Code):** 데이터의 무결성을 보장하기 위해 데이터와 비밀키, **시퀀스 번호**를 섞어 해시한 값입니다. 시퀀스 번호를 넣는 이유는 공격자가 패킷의 순서를 바꾸는 **재전송/순서 변경 공격(Replay/Re-order attack)**을 막기 위함입니다.
4.  **커넥션 종료 (Connection Closure):**
    *   데이터 전송이 끝났음을 알리기 위해 특별한 타입(Type 1)의 종료 메시지를 보냅니다. 이는 공격자가 TCP 핀(FIN) 메시지를 위조해 중간에 연결을 끊어버리는 **절단 공격(Truncation attack)**을 방지하기 위함입니다.

## 2. 방화벽 (Firewalls)

방화벽은 조직의 내부 네트워크를 외부 인터넷으로부터 격리하고, 특정 패킷은 통과시키고 특정 패킷은 차단하는 장치입니다. 보통 네트워크의 **게이트웨이** 위치에 설치됩니다.

### 2.1 방화벽의 종류
1.  **비상태 유지 패킷 필터링 (Stateless Packet Filtering):** 패킷의 헤더(IP, TCP/UDP 포트 번호 등)만 보고 룰(Rule)에 따라 독립적으로 판단합니다.
2.  **상태 유지 패킷 필터링 (Stateful Packet Filtering):** 모든 TCP 커넥션의 상태를 추적하여, 현재 유효한 커넥션과 관련된 패킷인지 판단합니다. 예를 들어, 외부에서 들어오는 응답 패킷이 이전에 내부에서 보낸 요청과 짝이 맞는지 확인합니다.
3.  **애플리케이션 게이트웨이 (Application Gateways):** 패킷 내부의 실제 데이터(애플리케이션 데이터)까지 검사하여 필터링합니다.

### 2.2 정책 및 ACL (Access Control Lists)
방화벽은 **ACL**이라는 룰 테이블을 위에서 아래로 순차적으로 적용합니다.
*   **예시 정책:** 외부 웹 접속 금지(포트 80 차단), 특정 서버 외의 TCP 연결 금지 등.
*   **현실적인 사례:** 대학교 전산망에서도 보안 사고 방지를 위해 SSH(22번 포트) 클라이언트 외부 접속을 막아두는 등의 정책을 사용하고 있습니다.

## 1. 보안의 4대 목표

| 목표 | 의미 | 위협 |
|------|------|------|
| **기밀성 (Confidentiality)** | 인가된 자만 내용 열람 가능 | 도청 (Eavesdropping) |
| **무결성 (Integrity)** | 전송 중 내용 변조 불가 | 중간자 공격 (Man-in-the-Middle) |
| **인증 (Authentication)** | 통신 상대방 신원 확인 | 위장 (Spoofing) |
| **가용성 (Availability)** | 서비스 정상 제공 | DDoS 공격 |

## 2. 암호화 기초

### 핵심 용어

```
평문 (Plaintext)  : 암호화 전 원본 데이터
암호문 (Ciphertext): 암호화된 데이터
암호화 (Encrypt)  : 평문 → 암호문
복호화 (Decrypt)  : 암호문 → 평문
키 (Key)          : 암호화·복호화에 사용하는 비밀값
```

### 암호화 수식
```
암호화: Ks(m) = c     (키 Ks로 메시지 m을 암호화 → 암호문 c)
복호화: Ks(c) = m     (키 Ks로 암호문 c를 복호화 → 평문 m)
```

---

## 3. 대칭키 암호화 (Symmetric Key)

### 개념
```
송신자와 수신자가 동일한 키 사용
암호화 키 = 복호화 키
```

```
Alice  ──[Ks(m)]──▶  네트워크  ──▶  Bob
         암호화                     Ks(c) 복호화
         (키: Ks)                   (키: Ks, 동일)
```

### 종류

| 알고리즘 | 키 길이 | 특징 |
|---------|--------|------|
| **DES** | 56비트 | 구식, 현재 취약 |
| **3DES** | 168비트 | DES 3번 적용, 느림 |
| **AES** | 128/192/256비트 | 현재 표준, 빠름 |
| RC4 | 가변 | SSL/WEP에서 사용 (현재 취약) |

### 블록 암호 vs 스트림 암호

| | 블록 암호 | 스트림 암호 |
|---|---|---|
| 방식 | 데이터를 블록 단위로 암호화 | 비트/바이트 단위로 실시간 암호화 |
| 예시 | AES, DES | RC4 |
| 사용 | 파일 암호화 | 실시간 스트리밍 |

### 대칭키의 핵심 문제: **키 배분 (Key Distribution)**
```
Alice와 Bob이 처음 만날 때 Ks를 어떻게 안전하게 공유?
→ 네트워크로 전송하면 도청 위험
→ 해결책: 공개키 암호화로 대칭키 전달
```

---

## 4. 공개키 암호화 (Public Key / 비대칭키)

### 개념
```
키가 2개: 공개키(Public Key) + 개인키(Private Key)
공개키로 암호화 → 개인키로만 복호화
개인키로 서명   → 공개키로 검증
```

```
Bob의 공개키: K+B  (모두에게 공개)
Bob의 개인키: K-B  (Bob만 보유)

Alice가 Bob에게 보낼 때:
  암호화: K+B(m) = c       (Bob의 공개키로 암호화)
  복호화: K-B(c) = m       (Bob의 개인키로만 복호화)
```

### RSA 알고리즘

```
원리: 큰 수의 인수분해가 매우 어렵다는 수학적 사실 활용

키 생성:
1. 큰 소수 p, q 선택 (각 1024비트 이상)
2. n = p × q
3. e, d 계산 → 공개키(n, e), 개인키(n, d)

암호화: c = m^e mod n
복호화: m = c^d mod n
```

**RSA 특성:**
```
K-B(K+B(m)) = m  (공개키 암호화 → 개인키 복호화)
K+B(K-B(m)) = m  (개인키 서명   → 공개키 검증)
```

### 대칭키 vs 공개키 비교

| | 대칭키 | 공개키 |
|---|---|---|
| 키 수 | 1개 (공유) | 2개 (공개+개인) |
| 속도 | 빠름 | 느림 (100~1000배) |
| 키 배분 문제 | 있음 | 없음 |
| 사용 | 대용량 데이터 암호화 | 키 교환, 서명 |
| 예시 | AES | RSA |

### 실제 사용 방식 (하이브리드)
```
1. 공개키 암호화로 대칭키(세션키) 안전하게 교환
2. 이후 데이터는 대칭키로 빠르게 암호화
→ 공개키의 안전성 + 대칭키의 속도 모두 확보
(SSL/TLS가 이 방식 사용)
```

## 5. 메시지 무결성 (Message Integrity)

### 문제 상황
```
Alice → Bob으로 메시지 전송 중
Trudy(공격자)가 메시지를 변조할 수 있음
→ Bob은 메시지가 변조됐는지 어떻게 알까?
```

### 암호화 해시 함수 (Cryptographic Hash)

```
H(m) = 고정 길이 해시값 (체크섬)

특성:
1. 단방향: H(m)으로 m을 역산 불가
2. 충돌 저항: 다른 m'에서 H(m) = H(m')이 나오기 극도로 어려움
3. 고정 길이: 입력 크기에 무관하게 항상 같은 길이 출력
```

| 알고리즘 | 출력 길이 | 상태 |
|---------|---------|------|
| MD5 | 128비트 | 취약 (사용 비권장) |
| SHA-1 | 160비트 | 취약 (사용 비권장) |
| **SHA-256** | 256비트 | 현재 표준 |
| SHA-3 | 가변 | 최신 표준 |

### MAC (Message Authentication Code)

```
목적: 무결성 + 인증 동시 제공

방식:
1. Alice와 Bob이 비밀키 s 공유
2. Alice: 메시지 m + s를 합쳐 해시 → MAC = H(m+s)
3. Alice가 m과 MAC 함께 전송
4. Bob: 받은 m과 자신이 가진 s로 H(m+s) 계산
5. 계산값 = 받은 MAC이면 → 무결성·인증 확인

공격자 Trudy는 s를 모르므로 MAC 위조 불가
```

## 6. 디지털 서명 (Digital Signature)

### 개념
```
공개키 암호화를 인증·무결성에 활용
개인키로 서명 → 공개키로 검증
→ 서명자만 개인키 보유이므로 부인 방지 가능
```

### 동작 방식

```
[서명 생성 - Alice]
1. 메시지 m 해시 → H(m)
2. 개인키로 해시 암호화 → K-A(H(m)) = 서명

Alice가 전송: m + K-A(H(m))

[서명 검증 - Bob]
1. Alice의 공개키로 서명 복호화 → K+A(K-A(H(m))) = H(m)
2. 받은 m을 직접 해시 → H(m)
3. 두 값 비교 → 일치하면 무결성·인증·부인방지 모두 확인
```

### 전체 메시지를 서명하지 않는 이유
```
RSA 서명은 느리다
→ 메시지 전체가 아닌 해시값(H(m))만 서명
→ 속도 문제 해결 + 보안 유지
```

### MAC vs 디지털 서명

| | MAC | 디지털 서명 |
|---|---|---|
| 키 | 대칭키 (공유) | 비대칭키 (공개+개인) |
| 부인방지 | ❌ | ✅ |
| 속도 | 빠름 | 느림 |
| 사용 | HMAC, TLS MAC | 인증서, 코드서명 |

## 7. 인증서·PKI

### 공개키 인증 문제
```
Bob의 공개키가 K+B라고 하는데, 진짜 Bob의 것인지 어떻게 확인?
→ 공격자 Trudy가 "나도 Bob이야, 내 공개키는 K+T야" 속일 수 있음
→ 해결책: 신뢰할 수 있는 제3자(CA)가 공개키 보증
```

### CA (Certificate Authority, 인증기관)

```
CA = 공개키의 신뢰성을 보증하는 기관
예: DigiCert, Let's Encrypt, Comodo

CA 역할:
1. 신원 확인 후 공개키에 CA의 디지털 서명 첨부
2. 이것이 = 인증서 (Certificate)
```

### 인증서 구조

```
인증서 내용:
┌─────────────────────────────────┐
│ 소유자 정보 (CN, 도메인명 등)    │
│ 소유자 공개키                    │
│ 발급 CA 정보                    │
│ 유효기간 (시작일~만료일)         │
│ 일련번호                        │
│ CA의 디지털 서명 ← 핵심         │
└─────────────────────────────────┘
```

### 인증서 검증 흐름

```
1. 서버가 인증서 전송
2. 클라이언트가 CA의 공개키로 서명 검증
3. 검증 성공 → 서버 공개키 신뢰
4. 서버 공개키로 통신 시작
```

### 인증서 체인 (Chain of Trust)

```
Root CA (최상위, 브라우저에 내장)
  └─ Intermediate CA (중간 CA)
       └─ End Entity Certificate (실제 서버 인증서)

브라우저는 Root CA 공개키를 미리 내장
→ 체인을 따라 올라가 Root CA 검증으로 전체 신뢰
```

## 8. SSL/TLS

### 개요
```
SSL  : Secure Sockets Layer (구버전, 취약)
TLS  : Transport Layer Security (현재 표준)
HTTPS = HTTP + TLS

위치: Application Layer와 Transport Layer 사이
역할: 기밀성·무결성·인증 모두 제공
```

### TLS Handshake 상세 흐름

```
[1단계 - ClientHello]
클라이언트 → 서버
- 지원하는 TLS 버전 (TLS 1.2, 1.3)
- 지원하는 암호화 방식 목록 (Cipher Suite)
- 랜덤값 R_c

[2단계 - ServerHello]
서버 → 클라이언트
- 선택한 TLS 버전
- 선택한 암호화 방식 (Cipher Suite)
- 랜덤값 R_s
- 서버 인증서 (공개키 포함)

[3단계 - 인증서 검증]
클라이언트:
- CA 서명 검증으로 서버 공개키 신뢰
- Pre-master Secret 생성

[4단계 - 키 교환]
클라이언트:
- Pre-master Secret을 서버 공개키로 암호화 → 전송
서버:
- 개인키로 복호화 → Pre-master Secret 획득

[5단계 - 세션키 생성]
양측 모두:
Master Secret = PRF(Pre-master Secret, R_c, R_s)
→ Master Secret에서 세션키 4개 생성
  (클라이언트 암호화키, 서버 암호화키,
   클라이언트 MAC키, 서버 MAC키)

[6단계 - Finished]
양측이 Finished 메시지 교환
→ 이후 세션키로 대칭 암호화 통신 시작
```

### TLS 1.3 개선점

```
TLS 1.2: Handshake 2 RTT
TLS 1.3: Handshake 1 RTT (절반으로 단축)
          취약한 암호화 방식 제거
          Forward Secrecy 기본 적용
```

### Cipher Suite 읽는 법

```
예: TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384

TLS           : 프로토콜
ECDHE         : 키 교환 알고리즘 (타원곡선 디피-헬만)
RSA           : 인증 알고리즘
AES_256_GCM   : 대칭키 암호화 (AES 256비트, GCM 모드)
SHA384        : MAC 해시 알고리즘
```

## 9. 이메일 보안

### 위협
```
- 도청: 이메일 내용 열람
- 위장: 발신자 주소 위조 (Spoofing)
- 변조: 내용 수정
```

### PGP (Pretty Good Privacy)

```
이메일 암호화·서명 표준

[암호화 + 서명 과정]
1. Alice가 세션키 Ks 생성
2. 메시지를 Ks로 대칭 암호화 → 빠름
3. Ks를 Bob의 공개키로 암호화
4. 해시로 서명 생성 (Alice 개인키 사용)
5. 전송: 암호화된 메시지 + 암호화된 Ks + 서명

[복호화 + 검증 과정]
1. Bob이 자신의 개인키로 Ks 복호화
2. Ks로 메시지 복호화
3. Alice의 공개키로 서명 검증
```

## 10. 네트워크 계층 보안 (IPSec)

### 개념
```
IP 계층에서 보안 제공
모든 IP 패킷을 암호화·인증
VPN 구현에 사용
```

### 두 가지 모드

| | 전송 모드 (Transport) | 터널 모드 (Tunnel) |
|---|---|---|
| 암호화 범위 | IP 페이로드만 | IP 패킷 전체 |
| IP 헤더 | 그대로 노출 | 새 IP 헤더 추가 |
| 사용 | 호스트 간 통신 | VPN (게이트웨이 간) |

### IPSec 프로토콜 2가지

**AH (Authentication Header)**
```
- 인증·무결성만 제공
- 암호화 없음
- IP 헤더 포함 전체 무결성 검증
```

**ESP (Encapsulating Security Payload)**
```
- 암호화 + 인증·무결성 모두 제공
- 실제로 대부분 ESP 사용
- VPN에서 주로 사용
```

### VPN 동작 방식

```
[본사 네트워크] ─── [VPN 게이트웨이] ─── 인터넷 ─── [VPN 게이트웨이] ─── [지사 네트워크]

패킷이 게이트웨이 통과 시:
원본 패킷 전체를 ESP로 암호화
+ 새 IP 헤더 추가 (터널 모드)
→ 인터넷 구간에서 내용 완전 차단
```

## 11. 방화벽 (Firewall)

### 개념
```
내부 네트워크와 외부(인터넷) 사이에서
패킷을 필터링하는 보안 장비·소프트웨어

목적:
- 허가되지 않은 접근 차단
- 내부에서 외부로 나가는 트래픽 제어
- 로그 기록
```

### 방화벽 종류 3가지

**① 패킷 필터링 (Packet Filtering)**
```
계층: 3~4계층 (IP·TCP 헤더 검사)
기준: IP 주소, 포트, 프로토콜

예시 규칙:
┌────────────┬────────┬──────────┬────────┐
│ 출발지 IP  │ 포트   │ 방향     │ 동작   │
├────────────┼────────┼──────────┼────────┤
│ Any        │ 80     │ 외부→내부│ 허용   │
│ Any        │ 443    │ 외부→내부│ 허용   │
│ Any        │ Any    │ 외부→내부│ 차단   │
└────────────┴────────┴──────────┴────────┘

장점: 빠름
단점: 상태 추적 불가, 우회 가능
```

**② 상태 추적 방화벽 (Stateful Inspection)**
```
계층: 3~4계층 + 연결 상태 추적
상태 테이블로 TCP 연결 흐름 추적

예: 내부에서 시작된 TCP 연결만 응답 허용
    외부에서 시작된 연결은 차단

장점: 패킷 필터링보다 정교함
단점: 애플리케이션 내용 검사 불가
```

**③ 애플리케이션 게이트웨이 (Proxy Firewall)**
```
계층: 7계층 (애플리케이션)
HTTP/FTP/DNS 내용까지 검사

예: 특정 URL 차단, SQL Injection 탐지

장점: 가장 정교함
단점: 느림, 비용 높음
```

### DMZ (Demilitarized Zone)

```
인터넷 ─── [외부 방화벽] ─── DMZ ─── [내부 방화벽] ─── 내부 네트워크
                               │
                          웹서버·메일서버
                          (외부 접근 허용)

DMZ: 외부에서 접근 가능하지만 내부망과 분리된 구간
→ 웹서버 침해돼도 내부망 보호
```

## 12. IDS / IPS

| | IDS | IPS |
|---|---|---|
| 이름 | 침입 탐지 시스템 | 침입 방지 시스템 |
| 동작 | 탐지 후 경보 | 탐지 후 차단 |
| 위치 | 네트워크 미러링 | 인라인 (트래픽 직접 처리) |
| 오탐 영향 | 경보만 | 정상 트래픽 차단 위험 |

### 탐지 방식

**시그니처 기반 (Signature)**
```
알려진 공격 패턴 DB와 비교
장점: 정확도 높음
단점: 새로운 공격(Zero-day) 탐지 불가
```

**이상 탐지 (Anomaly)**
```
정상 트래픽 패턴 학습 후 이상 감지
장점: 새로운 공격 탐지 가능
단점: 오탐(False Positive) 높음
```

## 13. 주요 네트워크 공격 유형

### DoS / DDoS

```
DoS  : 단일 공격자 → 대상 서버 과부하
DDoS : 다수 봇넷 → 대상 서버 과부하 (훨씬 강력)

종류:
- SYN Flood: 3-way Handshake SYN만 대량 전송
             서버가 반연결 상태 유지 → 자원 고갈
- UDP Flood: 대량 UDP 패킷으로 대역폭 소진
- HTTP Flood: 정상처럼 보이는 HTTP 요청 대량 전송
```

### 스니핑 (Sniffing)
```
네트워크 트래픽 도청
Wireshark 같은 도구로 패킷 캡처
→ 대응: 암호화 (HTTPS, TLS)
```

### 스푸핑 (Spoofing)
```
IP Spoofing : 출발지 IP 주소 위조
ARP Spoofing: ARP 응답 위조 → 트래픽 가로채기
DNS Spoofing: 가짜 DNS 응답 → 피싱 사이트로 유도
```

### 중간자 공격 (Man-in-the-Middle, MITM)
```
Alice ──▶ [Trudy] ──▶ Bob
         (가로채서 변조)

대응: 인증서로 상대방 신원 확인 (TLS)
```

### 피싱 (Phishing)
```
합법적인 사이트처럼 위장
→ 사용자가 직접 개인정보·비밀번호 입력
대응: EV 인증서 확인, 2FA
```

### SQL Injection
```
입력값에 SQL 문법 삽입 → DB 무단 접근
예: username = ' OR '1'='1
대응: 입력값 검증, Prepared Statement
```

### XSS (Cross-Site Scripting)
```
악성 스크립트를 웹페이지에 삽입
→ 피해자 브라우저에서 실행
대응: 입력값 이스케이프, CSP 헤더
```

## 14. 무선 네트워크 보안

### WEP (취약 — 사용 금지)
```
RC4 기반, 고정 키 사용
IV(초기화 벡터)가 24비트로 짧음
→ IV 재사용 → 키 역산 가능 (수 분 내 해킹)
현재 완전히 폐기
```

### WPA / WPA2

| | WPA | WPA2 |
|---|---|---|
| 암호화 | TKIP (RC4 개선) | AES-CCMP |
| 상태 | 취약 (폐기 권장) | 현재 표준 |
| 키 교환 | 4-way Handshake | 4-way Handshake |

### WPA2 4-way Handshake 취약점
```
KRACK (Key Reinstallation Attack, 2017):
핸드셰이크 메시지 재전송으로 키 재설치 유도
→ 암호화 우회 가능
대응: WPA3 전환 or 펌웨어 패치
```

### WPA3 (최신 표준)
```
SAE(Simultaneous Authentication of Equals) 사용
→ 오프라인 사전 공격 방지
Forward Secrecy 지원
개방형 네트워크도 암호화 (OWE)
```

### 무선 보안 위협

```
Evil Twin Attack (악성 AP):
  합법 AP와 동일한 SSID의 가짜 AP 설치
  → 사용자 연결 유도 → 트래픽 가로채기
  대응: VPN 사용, HTTPS 확인

Deauth Attack:
  Deauthentication 프레임 위조·대량 전송
  → 클라이언트 강제 연결 해제
  대응: 802.11w (관리 프레임 보호)

WPS PIN 공격:
  WPS 8자리 PIN 브루트포스
  대응: WPS 비활성화
```

## 15. 면접 핵심 Q&A

**Q. HTTPS가 HTTP와 다른 점?**
> HTTP는 평문 전송으로 도청·변조 가능. HTTPS는 TLS 암호화로 기밀성·무결성·인증 모두 제공. 443 포트 사용.

**Q. 대칭키와 비대칭키 차이?**
> 대칭키는 동일한 키로 암호화·복호화, 빠르지만 키 배분 문제. 비대칭키는 공개키·개인키 쌍, 느리지만 키 배분 안전. 실제로는 하이브리드 방식 사용.

**Q. TLS Handshake 과정 설명?**
> ClientHello → ServerHello + 인증서 → 인증서 검증 + Pre-master Secret 교환 → 세션키 생성 → Finished. 이후 대칭키로 통신.

**Q. 방화벽과 IDS 차이?**
> 방화벽은 사전에 정의된 규칙으로 트래픽 차단. IDS는 트래픽 분석해서 침입 탐지·경보. IPS는 탐지 + 차단까지.

**Q. 공개키 기반 인증서가 필요한 이유?**
> 공개키 자체는 누구든 만들 수 있어서 진짜인지 확인 불가. CA가 신원 확인 후 서명해주는 인증서로 공개키 신뢰성 보증.

**Q. VPN이란?**
> 공중 인터넷 위에 암호화된 터널 생성. IPSec ESP로 패킷 전체 암호화. 원격 근무자가 내부망에 안전하게 접근하는 데 사용.

**Q. SQL Injection 방어법?**
> Prepared Statement(파라미터 바인딩), 입력값 검증, 최소 권한 DB 계정, WAF 적용.

**Q. DDoS와 DoS 차이?**
> DoS는 단일 공격자. DDoS는 수천~수만 대의 봇넷으로 동시 공격. 규모와 차단 어려움이 다름.

**Q. WEP이 취약한 이유?**
> IV가 24비트로 짧아 재사용 빈번. RC4 취약점과 결합해 수 분 내 키 해독 가능. 현재 완전 폐기.

**Q. 디지털 서명의 3가지 기능?**
> 인증(발신자 신원 확인), 무결성(내용 변조 탐지), 부인방지(서명 사실 부인 불가).
