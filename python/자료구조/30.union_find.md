# 📂 [강의 자료] 유니온 파인드(Union-Find) 자료구조의 이해

**유니온 파인드(Union-Find)** 또는 **서로소 집합(Disjoint Set)** 자료구조는 여러 개의 원소들이 겹치지 않는 집합들로 나뉘어 있을 때, 이 집합들에 대한 연산을 효율적으로 제공하기 위해 설계되었습니다.

## 1. 집합과 유니온 파인드의 기초
*   **집합(Set)의 특징**: 집합 내의 키값들은 중복되지 않아야 합니다.
*   **기본 연산**: 파이썬의 `set` 자료구조는 멤버십 확인, 합집합, 교집합, 차집합 등을 효율적으로 제공하지만, 유니온 파인드 구조는 그중 **멤버십(Find)과 합집합(Union)** 연산에 집중하여 성능을 극대화합니다.

## 2. 유니온 파인드의 3가지 핵심 연산
1.  **`make_set(x)`**: 원소 `x` 하나만을 포함하는 새로운 집합을 만듭니다. 이 연산은 상수 시간 **$O(1)$**에 가능합니다.
2.  **`find(x)`**: 원소 `x`가 속한 집합을 찾아 그 집합의 **대표값(Representative)**을 반환합니다. 이를 통해 두 원소가 같은 집합에 속해 있는지 확인할 수 있습니다.
3.  **`union(x, y)`**: 원소 `x`가 속한 집합과 `y`가 속한 집합을 하나의 집합으로 합칩니다. 만약 이미 같은 집합에 속해 있다면 아무 작업도 하지 않습니다.

**목표 성능**: `find`와 `union` 연산을 **$O(\log n)$** 시간 내에 처리하는 것을 1차적 목표로 합니다.

---

## 3. 구현 방식 1: 양방향 연결 리스트 (비효율적)
하나의 집합을 하나의 원형 양방향 연결 리스트로 표현하는 방식입니다.
*   **작동 원리**: `find(x)` 연산 시 해당 노드에서 링크를 계속 따라가 헤드(더미) 노드를 찾아 이를 대표값으로 리턴합니다.
*   **한계**: 최악의 경우 `find` 연산 시 $n$개의 노드를 모두 따라가야 하므로 **$O(n)$**이 걸립니다. `union` 또한 `find`를 기반으로 하기에 동일하게 **$O(n)$**이 소요되어 비효율적입니다.

---

## 4. 구현 방식 2: 트리(Tree) 구조 (효율적)
각 노드가 자신의 **부모를 가리키는 링크**만을 가지는 트리 형태로 집합을 표현합니다.

### 4.1 구조적 특징
*   **링크 방향**: 일반적인 트리와 달리 자식을 가리키는 링크는 없으며, 오직 **부모 노드**만 가리킵니다.
*   **루트 노드**: 루트 노드는 부모가 없으므로 **자기 자신**을 가리키도록 설정하여 식별합니다.

### 4.2 연산 로직
*   **`find(x)`**: 노드 `x`에서 부모 링크를 타고 계속 올라가서 자기 자신을 가리키는 **루트 노드**를 찾아 반환합니다. 이 연산의 수행 시간은 트리의 **높이($h$)**에 비례합니다.
*   **`union(x, y)`**: 
    1.  `x`의 루트(대표값) `v`와 `y`의 루트 `w`를 각각 찾습니다.
    2.  두 루트가 다르다면, **한쪽 루트의 부모를 다른 쪽 루트로 설정**하여 두 트리를 하나로 합칩니다.
    3.  예를 들어, `w.parent = v`로 설정하면 `w`를 포함한 서브트리 전체가 `v` 집합의 일부가 됩니다.

---

## 5. 향후 과제: 높이 관리와 랭크(Rank)
트리 구조를 사용할 때 트리가 한쪽으로 길게 치우치면 성능이 저하됩니다. 이를 막기 위해 **랭크(Rank)**라는 값을 각 노드에 부여하고, `union` 시 랭크가 낮은 트리를 랭크가 높은 트리 아래에 붙이는 규칙을 사용합니다. 구체적인 랭크 활용법은 다음 강의에서 상세히 다루게 됩니다.

# 📂 [강의 자료] 유니온 파인드(Union-Find): 효율적인 합치기와 성능 분석

이전 강의에서 다룬 트리 구조 기반의 유니온 파인드를 더욱 최적화하기 위한 **유니온 바이 랭크(Union by Rank)** 전략과 그에 따른 시간 복잡도 분석을 다룹니다.

## 1. 효율적인 유니온 전략: Union by Height (Rank)

두 집합을 합칠 때(Union), 트리의 높이가 불필요하게 커지는 것을 방지하기 위해 **높이가 낮은 트리를 높이가 높은 트리 밑에 자식으로 연결**하는 전략을 사용합니다.

*   **동작 원리**:
    *   두 트리의 높이가 다르면, **높이가 더 큰 루트가 새로운 루트**가 됩니다. 이 경우 전체 트리의 높이는 변하지 않습니다.
    *   두 트리의 높이가 같으면, 어느 쪽이든 부모가 되어도 상관없으나 **합쳐진 후 전체 트리의 높이는 1 증가**하게 됩니다.
*   **용어 정의 (Rank)**: 각 노드에서 자신을 루트로 하는 트리의 높이를 저장하는 멤버를 **랭크(Rank)**라고 부릅니다.

## 2. 주요 연산의 구현 (Pseudo-code)

### 2.1 노드 및 초기화
각 노드는 부모를 가리키는 `parent` 링크와 높이 정보를 담은 `rank`를 가집니다. 처음에는 자기 자신을 부모로 설정하며 랭크는 0입니다.

```python
class Node:
    def __init__(self, key):
        self.key = key
        self.parent = self  # 루트는 자기 자신을 가리킴
        self.rank = 0       # 초기 높이는 0
```

```python
def makeset(x):
    return Node(x)
```

### 2.2 Find 연산
재귀적으로 혹은 반복문을 통해 부모 링크를 따라 올라가 최상위 루트 노드를 찾습니다.

```python
def find(x):
    while x.parent != x: # 루트에 도달할 때까지(루트가 아니라면)
        x = x.parent
    return x
```

### 2.3 Union 연산 (Rank 기준)
두 원소의 루트를 찾은 뒤, 랭크가 더 높은 쪽을 부모로 삼습니다.

```python
def union(x, y):
    v = find(x)
    w = find(y)
    if v != w: # 서로 다른 집합일 때만 합침
        if v.rank < w.rank:
            v.parent = w
        elif v.rank > w.rank:
            w.parent = v
        else: # 랭크가 같으면 한쪽을 부모로 정하고 랭크 1 증가
            v.parent = w
            w.rank += 1
```

## 3. 성능 분석: 왜 $O(\log n)$인가?

랭크 기반의 유니온 방식을 사용하면 $n$개의 노드를 가진 트리의 높이 $h$는 항상 **$\log_2 n$ 이하**로 유지됩니다.

*   **증명 아이디어**: 트리의 높이가 1 증가하려면 동일한 높이를 가진 두 트리가 합쳐져야 하며, 이때 전체 노드 수는 최소 2배가 됩니다.
*   높이가 $h$인 트리가 되기 위한 최소 노드 개수 $n_h$는 $2^h$와 같으므로, 결과적으로 **$h \le \log_2 n$**이 성립합니다.
*   따라서 `find`와 `union` 연산의 수행 시간은 트리의 높이에 비례하는 **$O(\log n)$**이 됩니다.

## 4. 요약 및 향후 과제
*   **핵심**: 높이가 낮은 트리를 높은 트리 아래에 붙임으로써 탐색 경로를 짧게 유지합니다.
*   **성능**: $O(\log n)$의 효율적인 연산을 보장합니다.
*   **더 나아가기**: `find` 연산 시 방문하는 노드들이 루트를 직접 가리키도록 업데이트하는 **경로 압축(Path Compression)** 기법을 사용하면 성능을 더욱 향상시킬 수 있습니다.
